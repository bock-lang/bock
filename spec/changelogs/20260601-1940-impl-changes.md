# §18.3.1 core.time — `Int64` storage prose reconciled with `Int` surface type

**Date:** 2026-06-01
**Affects:** §18.3.1 (`core.time`, `Duration`)
**Type:** clarification

> **Impl-alignment clarification only — no normative change.** This adds a
> single parenthetical to the §18.3.1 `Duration` prose; no signature, behavior,
> range, or type rule changes. Tracked as **Q-time-int64**.

## Change

§18.3.1 describes `Duration` as "Internally stored as `Int64` nanoseconds,
giving a range of approximately ±292 years", while the Bock-surface signatures
in the same section (the `Duration.*` constructors and the `as_nanos` /
`as_millis` / `as_seconds` accessors) take and return `Int`. A reader could
mistake this for a contradiction or infer a distinct `Int64` surface type.

The clarification states explicitly that the §18.3.1 surface type is `Int`,
which is `i64`-backed and therefore carries the full `Int64` range the storage
prose describes — there is no separate `Int64` surface type in these
signatures. The `Int64` mention remains, now scoped to storage width.

## Rationale

The implementation is internally consistent and unchanged: `Duration`
constructors accept `Int` (the interpreter's `expect_int` / `Value::Int`,
exercised by the `exec_core_time_duration` cross-target fixture with plain
integer literals), and the value is stored as signed `i64` nanoseconds (the
±292-year range the spec already claims; see `bock-interp` `Value::Duration`
and the `as i64` lowering in each backend). The spec prose conflated the
i64 *storage width* with the *surface type*; this aligns the wording to what
ships. Reconciles **Q-time-int64**.

## Migration

None. No code change is required; existing `Duration` code already uses `Int`
at the surface. This is a documentation clarification of behavior that already
holds on all five targets.
