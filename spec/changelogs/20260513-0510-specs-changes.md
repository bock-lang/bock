# LSP extensions and interpreter debugger UI deferred to v1.x

**Date:** 2026-05-13
**Affects:** §20.3, §20.5
**Type:** deferral
**Part of:** D1+D2 spec alignment consolidation

## Summary

The four Bock-specific LSP extensions described in §20.3 (AI Context Panel, Target Preview, Capability Graph, Smart Completions) are deferred to v1.x. The built-in interpreter debugger UI described in §20.5 is deferred to v1.x. Basic LSP capabilities (completion, hover, go-to-definition, diagnostics) remain in v1 and are not affected by this change.

## Change

### §20.3 LSP Bock-specific extensions

Current §20.3 describes the Bock LSP as advertising four extensions beyond the standard LSP protocol: AI Context Panel, Target Preview, Capability Graph, Smart Completions. None of these are implemented in v1.

The basic LSP capabilities (standard LSP protocol features) work in v1. Only the four Bock-specific extensions are affected.

Add a leading note to the Bock-specific extensions subsection of §20.3:

> **Status:** The four Bock-specific LSP extensions (AI Context Panel, Target Preview, Capability Graph, Smart Completions) are planned for v1.x. v1 ships standard LSP capabilities (completion, hover, definitions, diagnostics). The extension surface is preserved here as design intent for v1.x.

### §20.5 Interpreter debugger

Current §20.5 describes a built-in interpreter debugger with a UI surface. Source maps (the lower-level debugger infrastructure) are covered by `--source-map` / `--no-source-map` flags from the prior CLI surface amendments and ship in v1. The debugger UI is the v1.x piece.

Add a leading note to §20.5:

> **Status:** Source map generation (`--source-map` / `--no-source-map`) ships in v1 and enables external debugger integration through standard target-specific tooling (Node.js inspector, py-spy, rust-gdb, etc.). The built-in interpreter debugger UI described below is planned for v1.x. The UI design is preserved here as v1.x intent.

The body text of §20.5 remains as the planned v1.x surface.

## Rationale

Both LSP extensions and debugger UI are tooling surfaces orthogonal to language v1. Deferring them does not affect what programs are valid Bock or how the language semantics work; it affects what tooling experience the language ships with.

The basic LSP surface (standard protocol) works today. The four Bock-specific extensions are quality-of-life additions that benefit from their own design pass when v1 stabilizes. The AI Context Panel in particular needs UX design work that has not been done.

Source maps in v1 give users a debugging path through external target-specific tooling. The built-in interpreter debugger UI is a Bock-native experience that adds value but is not essential for users who can debug at the target level.

## Structural Note

Both §20.3's Bock-specific extensions and §20.5's debugger UI are tooling concerns. There is a reasonable case for moving these out of the spec body into tooling documentation when the v1.x design pass happens. That is a structural choice to revisit later; this changelog defers the surfaces in place rather than restructuring.

## Implementation Notes

Category: **impl-defined**. The basic LSP capabilities continue to ship; the four Bock-specific extensions are not implemented and the spec now reflects that. Source maps ship today via the CLI flags from the prior surface amendments.
