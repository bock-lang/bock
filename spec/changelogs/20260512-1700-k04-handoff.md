**Recommended conversion approach:**

A single Claude Code session can sweep all 61 package files. The

> Update all packages in `playbook/packages/` to remove

## Risk Assessment

**Low risk.** The consolidation is purely organizational. Three
specific risks worth flagging:

1. **In-flight Claude Code sessions referencing section files.**
   Any session started before this consolidation that tries to
   read `playbook/specs/s##.md` will fail. Recommended: pause
   in-flight sessions, sync workspace, restart them. Phase 0 is
   complete; current work is on Item B Phase 1 which has not
   started yet, so the practical exposure is minimal.

2. **Package files with stale section-file references.** The
   conversion sweep (step 2 above) catches these. Until that
   sweep runs, packages with file-path references are broken.
   Schedule the sweep before starting any new package work.

3. **Project knowledge serving stale spec content.** If users
   query the Claude project before the project knowledge update,
   they may get responses citing section file paths that no
   longer exist. The fix is just updating the project knowledge
   in the UI — a few minutes of work.

None of these block the consolidation; they're sequencing
concerns for the rollout.

---

## Why Now

The triggering condition for consolidation: "the agentic-
implementation phase has matured enough that section files are
no longer load-bearing."

Evidence:
- Recent design-pass changelogs have all touched both `bock-spec.md`
  and the corresponding section files. Same content, double
  maintenance, drift risk every cycle.
- D1 documentation buildout would have required a perpetual
  mapping table between section files and spec sections.
- Section files are excerpts that duplicate authoritative spec
  content. When the spec is mature (it is), the excerpts are
  pure overhead.

Pre-1.0 means we have the churn budget to consolidate now without
breaking external users. Post-1.0 the same change would be more
disruptive.
