# §18.5 trait-language integration: enumeration expanded, conformance surface added

**Date:** 2026-05-14
**Affects:** §18.5
**Type:** clarification + addition
**Part of:** Pending items from D1+D2 spec alignment consolidation

## Summary

§18.5 currently states the trait-language integration claim in one paragraph with vague enumeration ("Add/Sub/etc. enable operator overloading"). The D1 alignment refresh flagged that these claims aren't verified end-to-end: the trait machinery exists in `bock-core` (`TraitDispatch` at `bock-core/src/lib.rs:34`), but no central conformance test exercises the language-level integrations.

This changelog keeps the §18.5 normative wording (the integrations are v1 commitments) and adds two pieces:

1. **Explicit enumeration** of what trait → language integration means. The "Add/Sub/etc." gets unpacked into a concrete set so impl chat knows exactly what's normative.
2. **Conformance test surface** specifying what each integration must demonstrate. Cross-target equivalence is the acceptance bar per the §20.4 architectural principle (Bock owns cross-target correctness).

The spec wording stays normative; the addition makes verifiability concrete.

## Change

### §18.5 — Trait-Language Integration (expanded)

Replace the current one-paragraph §18.5 with expanded enumeration and a conformance subsection:

> ### 18.5 — Trait-Language Integration
>
> Core traits opt types into language features. Implementing a trait on a user-defined type enables the corresponding syntactic form for values of that type. The following integrations are normative for v1:
>
> | Trait (from `core.*`) | Language feature enabled |
> |-----------------------|--------------------------|
> | `Equatable` | `==`, `!=` operators |
> | `Comparable` | `<`, `>`, `<=`, `>=` operators |
> | `Iterable` | `for x in collection` loop syntax |
> | `Displayable` | `${expr}` string interpolation |
> | `Add` | `+` binary operator |
> | `Sub` | `-` binary operator |
> | `Mul` | `*` binary operator |
> | `Div` | `/` binary operator |
> | `Mod` | `%` binary operator |
>
> The integration is bidirectional: a type's `impl` block declares trait conformance, and the compiler uses that conformance to permit the corresponding syntactic form for values of that type. A type without `impl Comparable for MyType` cannot use `<` on `MyType` values; the compiler rejects the code at type-check time with a "type does not implement Comparable" diagnostic.
>
> Trait conformance is checked at every site where the syntactic form is used: arithmetic expressions, comparison in `if` conditions, comparison in `match` guards, ordering in sorts, equality in pattern matching, iteration in `for` loops, interpolation in string expressions. The integration is uniform: there are no "Comparable for if-conditions only" partial conformances; declaring `impl Comparable for MyType` enables `<` on `MyType` everywhere the operator is valid.
>
> #### Conformance test surface
>
> Each integration in the table above has a corresponding conformance test in the v1 conformance suite. Per the §20.4 architectural principle (Bock owns cross-target correctness), conformance is verified by running the same test program across all shipping targets and confirming equivalent observable behavior.
>
> The conformance suite for §18.5 verifies, at minimum:
>
> - **Equatable**: a user-defined type implementing `Equatable` supports `==` and `!=` in expressions, `if` conditions, `match` guards, and pattern matching equality checks.
> - **Comparable**: a user-defined type implementing `Comparable` supports `<`, `>`, `<=`, `>=` in expressions, `if` conditions, `match` guards, and as ordering for sort operations. Comparable conformance implies Equatable conformance.
> - **Iterable**: a user-defined type implementing `Iterable` supports `for x in collection` iteration, with correct handling of `break`, `continue`, and exhaustion. Nested iteration over Iterable types must work.
> - **Displayable**: a user-defined type implementing `Displayable` supports `${expr}` string interpolation, producing the implementation's defined output. Interpolation in nested contexts (interpolating a Displayable inside another interpolation) must work.
> - **Numeric operator overloading**: a user-defined type implementing `Add`, `Sub`, `Mul`, `Div`, or `Mod` supports the corresponding binary operator with correct operator precedence, associativity, and type checking on mixed operand types.
>
> A conformance test passes when its program produces equivalent observable output on every shipping target (JavaScript, TypeScript, Python, Rust, Go for v1; Java, C++, C#, Swift when added in v1.x). A failure on any target is a transpilation bug (§20.4), not a user-code bug.

## Rationale

### Why keep the wording normative (not soften)

The D1 alignment refresh recommended "either add a §18.5 conformance test surface or soften the wording." Softening would mean changing claims like "`Comparable` enables `<`/`>`" into "the compiler may dispatch `<` through `Comparable` conformance." That's a meaningful weakening: it tells users that operator overloading via traits is best-effort rather than a language guarantee.

Operator overloading via traits is core to the language design. Bock declares types and their behaviors; the compiler dispatches operators through trait implementations. Softening this would erode a fundamental value proposition. The right path is to keep the commitment and verify it.

### Why explicit enumeration

The original "Add/Sub/etc." leaves the rest of the operator surface ambiguous. Does it include Mul/Div/Mod? Bitwise operators? Unary negation? Without explicit enumeration, the impl chat has to guess what's in v1 scope.

The enumeration above is the v1 surface: equality, ordering, iteration, interpolation, and the five binary numeric operators. Other potential trait-integrations (bitwise, unary, shift, comparison-via-three-way) are not in v1 scope. If they're added in v1.x, this enumeration gets extended via a new changelog.

### Why conformance test surface in spec

The §20.4 architectural principle (added by the §15 changelog at `20260514-0408-specs-changes.md`) states that Bock owns cross-target correctness. The §18.5 conformance tests operationalize that principle for trait integrations: same Bock program, equivalent behavior across all targets, verified by automated test.

Without a conformance test surface, "the integrations work" is a claim no one can audit. With one, v1 release becomes conditional on the conformance tests passing.

### Adding Equatable explicitly

The original §18.5 didn't mention Equatable, but Equatable is in §18.2's prelude as a core trait, and `==` and `!=` are operators that must dispatch through something. Including Equatable in §18.5's enumeration makes explicit what was implicit. This is a clarification, not a new commitment.

The Comparable-implies-Equatable relationship matches standard library conventions (in Rust, Comparable's bound; in Swift, Comparable inherits Equatable). Stating it explicitly avoids ambiguity about whether implementing Comparable alone enables == on the type.

## Implementation Notes

Category: **impl-must-update**.

**Required impl updates for v1:**

The trait machinery already exists; the gap is end-to-end verification. The paired implementation session:

1. **Audit existing integrations.** For each trait → operator/syntax pair in the §18.5 table, identify whether the integration is wired today. The impl chat's report noted this isn't centrally documented; the audit produces the missing documentation as a byproduct.

2. **Build the conformance test suite for §18.5.** Each integration gets a Bock program that:
   - Defines a user-defined type
   - Implements the relevant trait
   - Exercises the syntactic form (operator, for..in, interpolation)
   - Asserts the expected behavior
   - Runs cleanly on every shipping target (JS, TS, Python, Rust, Go)

3. **Wire missing integrations.** If the audit identifies a gap (for example, `<` works in expressions but not in `match` guards), wire it. Use the audit results to scope the work.

4. **Add the suite to release criteria.** v1 release is contingent on §18.5 conformance tests passing on all five shipping targets. The conformance suite goes into the existing test infrastructure (presumably `compiler/tests/conformance/` or equivalent; impl chat picks the exact location).

**Paired session acceptance criteria:**

- Conformance suite covers all nine integrations in §18.5 (Equatable through Mod)
- Each conformance test passes on JavaScript, TypeScript, Python, Rust, and Go
- Test failures produce diagnostics identifying which integration and which target failed
- The suite is run as part of `cargo test --workspace` (or whatever the release-readiness gate is)

If the audit uncovers that some integration is materially missing (e.g., Iterable's for..in works but break/continue interactions are broken), that's a separate impl task to file as part of the session output. The conformance suite documents the gap; the wiring fix lands as a separate commit.

**Scoping note:** This is paired-session work, not a single-changelog drop. The audit may surface integration gaps that take real implementation work to close. If the audit reveals that current impl ships with materially broken integrations, surface as OPEN before declaring v1 release-ready.

## Migration

For users (none yet, since v1 hasn't shipped): no migration. The integrations should work; if a user writes code that uses these patterns today and it works, it continues working. If it doesn't work today but should, the paired impl session fixes it before v1.

For the spec: the §18.5 expansion replaces a one-paragraph section with a fuller treatment. Anyone navigating the spec for trait-language semantics now finds the explicit enumeration and conformance surface in one place.
