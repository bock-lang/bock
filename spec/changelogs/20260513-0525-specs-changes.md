# Supported targets split into shipping vs planned

**Date:** 2026-05-13
**Affects:** §1.3
**Type:** clarification (no semantics change)
**Part of:** D1+D2 spec alignment consolidation

## Summary

§1.3 currently presents nine supported targets in a single list, all in present tense, with no indication that v1 ships five of them and four are roadmap items. The list is restructured into two clearly labeled groups: "Ships in v1" (JavaScript, TypeScript, Python, Rust, Go) and "Planned for v1.x" (Java, C++, C#, Swift). The change is purely descriptive; no language semantics are affected, and the v1.x list is non-binding intent rather than commitment.

## Change

Current §1.3 lists supported targets as:

> Bock transpiles to JavaScript, TypeScript, Python, Rust, Go, Java, C++, C#, and Swift.

(Approximate phrasing; exact §1.3 wording in the spec.)

Restructure into a two-group format:

> **Ships in v1.** Bock transpiles to five targets in v1:
> - JavaScript (Node.js and browser runtimes)
> - TypeScript (Node.js and browser runtimes with type emission)
> - Python (3.10+)
> - Rust
> - Go
>
> **Planned for v1.x.** Four additional targets are on the v1.x roadmap:
> - Java
> - C++
> - C#
> - Swift
>
> The v1.x list expresses target ambition and informs the design of the AIR and codegen architecture; it is not a v1 commitment. Users requiring these targets in v1 should expect to wait for v1.x.

## Rationale

The current §1.3 wording overstates what v1 ships. A reader sees "Bock transpiles to Java" and reasonably expects to use it in v1; the actual answer is "not yet."

Three options were considered:

1. **Drop the four targets from §1.3 entirely.** Conservative; removes the language's target ambition from the spec.
2. **Ship the four targets in v1.** Substantial codegen work; not realistic for v1.
3. **Split shipping vs planned.** Honest about state; preserves ambition; sets v1.x expectations.

Option 3 is the path. The target-agnostic positioning is core to Bock's value proposition, and §1.3 should reflect both what ships and what is planned without conflating them.

The "planned" wording is deliberately soft. "On the v1.x roadmap" is not a commitment to ship by a specific date; it states design intent. The codegen architecture (AIR, target profiles, capability gap synthesis) is designed to support the planned targets, which is the design-time consequence of the v1.x intent.

## Marketing Note

This change makes the "5 targets shipping, 4 planned" framing accurate for marketing copy. Marketing chats can use "nine supported targets" only with the "five shipping, four planned" qualifier; the unqualified "nine targets" framing overstates.

## Implementation Notes

Category: **impl-defined**. Current impl ships JS, TS, Python, Rust, Go; the spec change matches that state. The four planned targets remain forward work, scheduled per the v1.x roadmap.
