# Spec Changes — 2026-04-16 10:00

Addition of §10.8 — Adaptive Effect Handlers.

---

## Context

Discussion of language differentiators identified a natural
extension of Aura's effect system: AI-assisted runtime error
recovery that leverages the semantic context already carried by
`@context`, `@performance`, `@domain`, and `@security` annotations.

Three tiers were evaluated:
1. **Intelligent error recovery** — LLM selects from closed set
   of developer-defined recovery strategies (APPROVED)
2. **Runtime diagnostic analysis** — AI-enhanced error messages
   (deferred to tooling, not a language feature)
3. **Adaptive code proposals** — runtime-observed patterns
   generate PRs (deferred as design vision, not specced)

Only Tier 1 is specified. The key differentiator: no other
language gives error handlers access to structured semantic
metadata about what the code is trying to accomplish.

## Design Constraints

The following constraints are intentional and load-bearing:

**Closed-set selection.** The LLM never generates code, never
synthesizes behavior, never executes arbitrary output. It selects
from strategies the developer explicitly defined. This is
classification, not generation.

**AI not required for correctness.** If the AI provider is
unavailable, the handler falls back to the first strategy in
the list (developer's preferred default). The program runs
correctly without AI — the AI just makes recovery smarter.

**Production freezes to deterministic.** In production strictness,
all strategy selections are pinned. The adaptive handler becomes
functionally identical to a deterministic handler. Developers
tune in development, freeze for production. No runtime AI calls
in production unless explicitly opted in.

**No new syntax.** Adaptive handlers use the existing `handling`
block syntax (§10.3). `Network.adaptive(...)` is an associated
function call (§6.7) returning a handler implementation. The
grammar is completely unchanged.

## Spec Amendments

### §10.8 — Adaptive Effect Handlers — NEW
Inserted after §10.7 (Graduated Strictness), before §11
(Context System). Covers:

- Defining adaptive handlers via `Effect.adaptive()` combinator
- Closed-set constraint (never generates code)
- Context-aware strategy selection using annotations
- Strategy selection logging in decision manifest (§17.4)
- Strictness interaction (auto/logged/pinned)
- Fallback behavior when AI unavailable
- Built-in strategy combinators: `retry`, `use_cached`,
  `degrade`, `circuit_break`, `escalate`
- Custom strategies via `RecoveryStrategy[E, T]` trait
- `RecoveryContext` type providing error history, annotations,
  and timing information

### playbook/specs/s08-effects.md — EXPANDED
Added "Adaptive Effect Handlers" section summarizing the
closed-set selection model, key constraints, and code example.

## Implementation Implications

This is a Phase 5/6 feature (interpreter + core library):

**P5.5 (Effect handler runtime)** — The adaptive handler logic
(strategy selection dispatch, fallback behavior) would extend
the effect handler runtime. If already implemented, this is a
follow-up package.

**P6.x (new package)** — The built-in strategy combinators
(`retry`, `use_cached`, `degrade`, `circuit_break`, `escalate`)
and the `RecoveryStrategy` trait should be a new package in
Phase 6, dependent on P5.5 and the AI provider interface (§17.8).

**P9.x (codegen)** — Adaptive handler transpilation follows the
same parameter-passing strategy as regular handlers (§10.6).
The strategy selection logic is runtime code, not special syntax.

No changes to the current 61-package plan are required immediately.
The adaptive handler infrastructure can be added as a Phase 6
extension or a post-v1 package once the effect runtime is stable.
