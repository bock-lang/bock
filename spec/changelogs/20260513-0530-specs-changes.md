# Capability gap synthesis principle normative; table illustrative

**Date:** 2026-05-13
**Affects:** §17.6
**Type:** clarification (no semantics change for current impl)
**Part of:** D1+D2 spec alignment consolidation

## Summary

§17.6 currently presents a table of specific synthesis strategies per (construct, target) pair. The table reads as normative: codegen must implement these specific strategies. Current impl behaves as if the table were illustrative: codegen chooses synthesis based on target profile and AI provider Generate mode, not a locked enumeration.

This changelog reframes §17.6 to make the underlying principle normative (synthesis happens via Generate mode per §17.8 against the confidence threshold from §17.4) and the specific synthesis table illustrative (examples of how synthesis manifests, not contracts on the synthesis output).

## Change

### Normative principle (new in §17.6)

Add a normative principle at the top of §17.6, preceding the existing table:

> At capability gaps (constructs where the target lacks a direct equivalent), codegen invokes the AI provider's Generate mode (§17.8) with the target profile and surrounding context. Generate's output is accepted when its confidence meets or exceeds the threshold in §17.4 (default 0.75 with `--strict` raising to 0.90); below threshold, codegen falls back to the deterministic strategy described in the target profile or surfaces an unrecoverable capability gap to the user.
>
> The synthesis strategies in the table below are illustrative examples of how this principle manifests for common (construct, target) pairs. They are not normative: codegen may produce alternative syntheses that satisfy the confidence threshold and pass target profile verification. The table is informative for users tracking how their code is likely to be synthesized.

### Existing table reframed

Add a heading and intro to the existing table:

> ### Illustrative synthesis examples
>
> The table below shows common synthesis outcomes for representative (construct, target) pairs. Actual synthesis may differ when the AI provider produces an alternative that meets the confidence threshold.

The table content itself is unchanged. The reframing comes from the headings and the new normative principle above.

## Rationale

The current §17.6 table reads as a contract: for each (construct, target) pair, codegen produces the specific strategy listed. Reading it that way creates two problems:

1. **Constrains codegen evolution.** Synthesis strategies improve over time. Locking the spec to specific strategies prevents codegen from adopting better approaches without spec amendment.
2. **Undermines the AI provider value.** The whole point of Generate mode at capability gaps is that the AI provider can produce synthesis informed by target context. If specific strategies are locked, Generate is just executing a lookup table, not actually generating.

The principle is what matters: at capability gaps, codegen uses Generate mode against a confidence threshold. The specific outputs are downstream of that principle.

Making the principle normative locks in the language guarantee (synthesis happens; it is bounded by confidence; it falls back to deterministic if not confident enough). Making the table illustrative gives codegen freedom to evolve while keeping the spec useful as a guide.

This reframing also resolves the question raised by the D1 alignment refresh: "is §17.6 a conformance test surface?" Under the new framing:

- The **principle** is a conformance surface (codegen must invoke Generate at capability gaps; must respect confidence threshold; must fall back deterministically)
- The **table** is not a conformance surface (codegen may produce alternatives)

This matches what current impl does. The spec change documents the existing behavior accurately rather than mandating something different.

## Implementation Notes

Category: **impl-defined**. Current impl already behaves as the new framing describes: Generate mode is invoked at capability gaps with confidence threshold checking; the specific synthesis output varies. The spec change documents this existing behavior.

This is the cleanest kind of spec amendment: the impl was correct; the spec wording was misleading; the wording is now corrected.
