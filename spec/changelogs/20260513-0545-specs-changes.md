# Default parameter values: clarified for v1 with paired implementation session

**Date:** 2026-05-13
**Affects:** §6.1
**Type:** clarification (locks semantics for paired implementation)
**Part of:** D1+D2 spec alignment consolidation

## Summary

§6.1 illustrates default parameter values:

```bock
fn greet(name: String, greeting: String = "Hello") { ... }
greet("Alice")  // greeting defaults to "Hello"
```

The default-value syntax parses (the AST carries the default expression). Call-site default application does not work: call sites still require every parameter to be supplied. Surfaced by D2's example execution verification.

Default parameter values are a common ergonomic feature users expect from any modern language. The implementation gap is bounded (call-site resolution logic that substitutes defaults). This changelog ships default parameter values in v1 and locks the semantics so the paired implementation session has clear acceptance criteria.

## Change

§6.1 prose is unchanged in substance. The semantics governing default value evaluation are stated explicitly to give the paired impl session unambiguous acceptance criteria. Add a Semantics subsection to §6.1:

### Semantics (new subsection)

> **Evaluation timing.** Default value expressions are evaluated per-call at the call site, not once at function definition. Each invocation that omits the parameter evaluates the default expression fresh.
>
> ```bock
> fn append_log(msg: String, log: List<String> = List.new()) { ... }
> append_log("a")  // log is a fresh empty List
> append_log("b")  // log is a different fresh empty List
> ```
>
> This avoids the Python-style "mutable default argument" gotcha where the default is shared across calls.
>
> **Argument binding order.** Parameters are bound positionally first, then named arguments fill remaining parameters by name. Defaults apply to parameters that remain unbound after positional and named arguments are resolved.
>
> **Type checking.** The default expression must produce a value compatible with the parameter's type. Type checking happens at function definition; an incompatible default is a compile-time error.
>
> **Effect tracking.** If the default expression invokes effectful operations (`List.new()` allocates; a hypothetical `current_time()` would require the time capability), the effect is attributed to the call site that triggers the default, not the function definition. The function's effect signature reflects defaults conservatively.
>
> **Target codegen consistency.** All targets produce the same observable semantics. Targets with native default parameter support (JavaScript, Python) use the native form with care to match per-call evaluation. Targets without native defaults (Go) synthesize equivalent call-site checks.

## Rationale

Default parameters are table-stakes ergonomics for any modern language. Users coming from JavaScript, TypeScript, Python, Rust, Go, Swift all expect them. Shipping v1 without default parameter application would create a noticeable feature gap.

The implementation gap is bounded. The parser already accepts the syntax; the AST carries default expressions. What's missing is call-site resolution logic that substitutes defaults when parameters are omitted. This is mechanical work.

Three semantic questions decided by this changelog:

1. **Per-call evaluation** (not once-at-definition). Avoids the Python mutable-default-argument trap. Matches expected behavior for users coming from most modern languages.
2. **Positional-first binding order.** Positional arguments bind first; named arguments fill remaining parameters; defaults apply to parameters that remain unbound. Predictable and unambiguous.
3. **Effect attribution at call site.** Defaults that invoke effectful operations attribute the effect to the call site. The function's effect signature includes the union of declared effects and any effects from default expressions.

These decisions give the paired implementation session unambiguous acceptance criteria.

## Implementation Notes

Category: **impl-must-update**. The spec is correct as written for v1; current impl has a gap (defaults parse but don't apply at call sites).

**Paired implementation session required.** Scheduled to land before v1 release. Acceptance criteria:

- Call sites omitting parameters use the default expression for those parameters
- Default expressions evaluate per-call (verified by mutable-default test case)
- Positional + named + defaults bind in the order specified above
- Type checking of defaults happens at function definition
- All five shipping targets produce equivalent observable behavior (verified by conformance test)

The paired session should produce its own changelog at landing time noting completion and any impl-side discoveries.

If the paired session uncovers complexity beyond the bounded estimate (for example, a non-trivial interaction with generics or effect inference), surface as OPEN and consider falling back to deferral. The default disposition is ship; the fallback is defer.
