## Class A — Spec amendments (impl ahead of spec)

These flags and subcommands exist in the shipping CLI; §20.1 now
documents them.

### A-1. `bock build --strict`

Forces production strictness for the build regardless of project
default. Fails on unpinned build-scope decisions. Pairs with the
§10.7 graduated strictness model and §17.4 pinning requirement.

### A-2. `bock build --pin-all`

Pins every build-scope decision after a successful build. Pairs
with `--strict` for the develop → ship workflow: build with
`--pin-all`, commit the manifest, ship CI builds with `--strict`.

**§17.4 forward reference added.** The workflow this enables is
non-obvious from §20.1 alone. A new "Build workflow" paragraph in
§17.4 describes the develop → ship transition and points back at
§20.1.

### A-3. `bock build --source-map` / `--no-source-map`

Source map emission toggle. Default-on. Pairs with §20.5's source
map reference for debugging transpiled code.

### A-4. `bock pkg init` / `tree` / `list`

Standard package-manager verbs not previously enumerated in
§20.1. Orthogonal to the existing `add`/`remove`/`update`/
`audit`/`publish`/`search` set (some of those are deferred — see
C-3 below).

### A-5. `bock pkg cache`

Manages the tarball cache under `.bock/cache/`. Distinct from
`bock cache` (AI/decision/rule caches). Two different cache
surfaces with different lifecycles — the §20.1 entry calls this
out explicitly so users (and future spec readers) don't conflate
them.

## Class C — Spec deferrals (impl behind spec by intent)

These spec surfaces describe v1.x or post-v1 functionality.

### C-1. `bock test` cross-target flags

v1 ships `--filter`. `--target`, `--all-targets`, `--smart`,
`--coverage`, `--snapshot` defer to v1.x. Cross-target testing
pairs with Item B project mode work; smart targeting depends on
AIR change-impact analysis not yet built.

### C-2. `bock cache list` / `prune`

v1 ships `stats` + `clear`. `list` (enumerate entries) and
`prune` (age-based eviction) are conveniences for v1.x.

### C-3. `bock pkg update` / `audit` / `publish` / `search`

Each depends on infrastructure not yet built: stable resolver
invariants, security advisory feed, public registry + auth,
registry search API. Marked "ships alongside the public registry"
in §20.1; cross-reference to §19.

### C-4. `bock model list` / `install` / `use`

v1 ships `show` + `set` for remote provider configuration. Local
model management defers to v1.x alongside local provider support.

### C-5. `bock fix`

Auto-fix for lint warnings. Semantics (safety, idempotence,
source preservation) warrant a design pass before implementation.
Moved to "Planned for v1.x" in §20.1.

### C-6. `bock migrate`

AI-assisted import from other languages. Requires per-source-
language frontends plus AI pipeline reversal. Marked "Planned"
in §20.1.

### C-7. `bock target`

Top-level target management verb. v1's target customization
already happens via `[targets.<T>]` in `bock.project` and
`--target` on `bock build`. The top-level verb is reserved for
future use; §20.1 notes the v1 surface is the `[targets.<T>]`
config block.

### C-8. `bock ci` — REMOVED

Decision: remove, not defer. `bock ci` would be an alias for
`bock fmt --check && bock check && bock test`, which is already
expressible with the existing primitives. Deferring it creates a
hole in the spec someone will eventually try to fill, possibly
with semantics that don't match the composition. If the composed
pattern proves common in real usage, adding `bock ci` later as a
shorthand is cheap.

## Section Restructure

§20.1 was reorganized for readability. The command list is now
grouped by purpose with a "Planned for v1.x or later" trailing
section for the deferred surfaces. The previous flat list mixed
v1 and aspirational commands without distinction.

Three thematic groups remain:
- **Build and execute** — `new`, `build`, `run`, `check`,
  `test`, `fmt`, `repl`
- **Decision and rule manifest management** — `inspect`, `pin`,
  `unpin`, `override`, `cache`
- **Project lifecycle** — `promote`, `doc`, `pkg`, `model`

`bock fix`, `bock migrate`, `bock target` listed separately as
planned.
