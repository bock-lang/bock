# Module system: brace-less and aliasing use-forms deferred to v1.x

**Date:** 2026-05-13
**Affects:** §12.2
**Type:** deferral
**Part of:** D1+D2 spec alignment consolidation

## Summary

§12.2 illustrates three accepted forms for the `use` statement:

- Braced single-name: `use std.io.{print}`
- Brace-less single-name: `use std.io.print`
- Aliased: `use std.collections.HashMap as Map`

Implementation resolves only the braced form. The brace-less single-name form parses but does not resolve; the aliased form does not parse at all. Surfaced by D2's example execution verification.

The brace-less and aliased forms are deferred to v1.x. §12.2 examples standardize on the braced form. The brace-less form is removed from spec body (it adds spec surface without functional difference from braces); the aliased form is preserved with a Reserved marker.

## Change

### Brace-less single-name `use` removed from §12.2

Current §12.2 illustrates both `use std.io.print` and `use std.io.{print}` as equivalent. The brace-less form is ergonomic sugar that adds spec surface without functional difference.

Remove brace-less examples from §12.2. Standardize all single-name examples on the braced form:

```bock
use std.io.{print}
```

Add a note explaining the choice:

> §12.2 standardizes on the braced form for `use`. Single-name imports are written with braces (`use module.{name}`) for consistency with multi-name imports (`use module.{name, other}`). The brace-less form (`use module.name`) is not part of the v1 surface.

### Aliased `use` Reserved for v1.x

Current §12.2 illustrates `use module as alias`. The `as` keyword is not recognized in `use` position; no alias resolution path exists in v1.

Mark aliased imports Reserved for v1.x:

> **Reserved for v1.x: aliased imports.** The `use module as alias` form (`use std.collections.HashMap as Map`) is planned for v1.x. v1 supports braced imports without aliasing. Aliasing becomes important once a third-party package ecosystem creates naming conflicts; v1 ships with first-party stdlib only.

The aliased-form examples in §12.2 are caveated rather than removed, preserving the design intent for v1.x.

## Rationale

**Brace-less form.** The braced form covers the functionality. Brace-less single-name is sugar that saves two characters. The cost of supporting two forms (parser handling, documentation surface, user choice paralysis) outweighs the saving. Standardizing on braces is the cleaner v1 commitment.

**Aliased form.** Aliasing solves real naming conflicts but those conflicts arise from third-party packages with overlapping names. v1 ships without third-party packages; aliasing is premature. The Reserved marker lets v1.x add aliasing as a natural extension when an ecosystem develops.

## Implementation Notes

Category: **impl-defined** for both. Current impl resolves the braced form and rejects the others (brace-less parses but doesn't resolve; aliased doesn't parse). The spec changes align prose with existing impl state. No code change is required.

A small follow-up impl task: improve the diagnostic for brace-less use-statements to suggest the braced form (current behavior is silent non-resolution). This is a polish item, not blocking.
