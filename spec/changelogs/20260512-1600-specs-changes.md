## Background

Documentation buildout (D1) surfaced what the inventory matrix
labeled a "polarity difference (same capability)" between
`--context` (spec) and `--no-context` (impl). On verification,
they are different features with confusingly similar names:

- **Spec's `--context`** ŌĆö selective-check mode running only
  context-system validation (analogous to `--types`, `--lint`)
- **Impl's `--no-context`** ŌĆö diagnostic-rendering toggle hiding
  source-context snippets in error output

Users reading the spec today would reach for `--context` and
encounter `--no-context`, which does something else. Even after
disambiguation, the namespace collision (`--context` /
`--no-context`) makes both flags harder to discover and reason
about.

---

## Resolution

### Spec's selective-check flags ŌåÆ `--only=<aspect>` pattern

The three selective-check flags (`--types`, `--lint`, `--context`)
become one flag with documented aspect values:

```bash
bock check --only=types
bock check --only=context
bock check --only=types,context
bock check --only=types --only=context     # repeated form equivalent
```

**Semantics:**

- Omitted (`bock check`) ŌåÆ run all aspects (full check, default)
- Single value (`--only=types`) ŌåÆ run only that aspect
- Multiple values (comma-separated or repeated) ŌåÆ run those aspects
- Unknown aspect ŌåÆ error with the list of valid aspects

**Why this is better:**

1. One mental model ("only check these aspects") instead of three
   flags users discover separately
2. Multiple aspects compose naturally ŌĆö `--only=types,context`
   for a pre-commit hook that wants types + context but skips
   the heavier lint pass
3. Extending to new aspects later doesn't add new flags ŌĆö it
   adds a new valid value

**v1 aspect surface:**

- `types` ŌĆö type checking
- `context` ŌĆö context-system validation (┬¦11)

**v1.x aspect surface** (ships alongside `bock fix`):

- `lint` ŌĆö the lint pass exists in v1 (`bock check` runs it by
  default), but `--only=lint` ships when `bock fix` provides
  the ergonomic counterpart

### Impl's `--no-context` ŌåÆ `--brief`

The diagnostic-rendering toggle that hides source-context snippets
in error output renames from `--no-context` to `--brief`.

**Why:**

1. Eliminates the namespace collision with `--only=context`
2. Describes user intent ("give me a compact error summary")
   rather than implementation detail ("hide source context")
3. Pairs naturally with an implicit `--verbose` opposite if
   one is ever needed
4. Reads as a deliberate UX choice rather than a negation flag

**Default unchanged.** Source context is still shown by default;
`--brief` opts into the compact form.

---

## Why both renames vs. just renaming `--no-context`

The minimal change (rename only `--no-context` to `--brief`) would
eliminate the collision with a single edit. The `--only=<aspect>`
pattern is a separate ergonomic improvement that happens to be
made cheaper by the renaming pass.

The case for the double rename:

- Three selective-check flags (`--types`, `--lint`, `--context`)
  is genuinely worse than one selective-check flag with documented
  aspects. The pattern would have been worth doing on its own.
- Pre-1.0 churn budget. Bock has not shipped; users haven't
  embedded these flags in scripts. The cost of both renames now
  is lower than either rename later.
- The ┬¦20.1.1 aspect surface section becomes a single-stop
  reference for `bock check` invocation. Three flags would have
  required three documentation entries with overlapping prose.

The case against: more user-facing surface change in a single
batch. Acceptable in this case because the changes are
coordinated (eliminating one collision while improving
ergonomics) rather than independent breaking changes.

## Implementation Implications

Both renames are CLI surface changes in `bock-cli`. Approximate
scope:

1. **Remove** `--types`, `--lint`, `--context` from the
   `bock check` clap definition
2. **Add** `--only` parameter accepting comma-separated list
   with validation against the v1 aspect set
3. **Rename** `--no-context` to `--brief` in the `bock check`
   clap definition; update the help text to describe the user
   intent ("compact error output without source-context
   snippets") rather than the implementation
4. **Update** any internal usage in tests, CI scripts, examples,
   docs

Pre-1.0 means no migration shim is required, though the
implementation chat may choose to emit a one-cycle deprecation
warning for users running early builds if it's cheap.
