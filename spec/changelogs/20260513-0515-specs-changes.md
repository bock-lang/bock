# Effects spec alignment with Appendix A.3 + lambda handler deferral

**Date:** 2026-05-13
**Affects:** §10.3, §10.4, §10.6
**Type:** clarification + deferral
**Part of:** D1+D2 spec alignment consolidation

## Summary

Three §10 effects items resolved together:

- **§10.3 Layer 3** currently describes three-layer effect handler resolution; Layer 3 references the `[effects]` table that Appendix A.3 has marked Reserved. The two need to align: §10.3 marks Layer 3 Reserved for v1.x.
- **§10.4 lambda handlers** (surfaced by D2's example execution verification): `Effect.handler(...)` lambda-form handlers don't parse. Record-shaped handlers (`with Logger as { log: (msg) => print(msg) }`) work today. The lambda form defers to v1.x.
- **§10.6 effect erasure** is worded permissively but reads as a v1 commitment; the wording softens to "MAY apply" with a forward reference.

The three are bundled because they all touch §10 effects semantics and resolve as a coordinated cleanup.

## Change

### §10.3 Layer 3 alignment

Current §10.3 describes effect handler resolution in three layers:

1. Function-local handlers
2. Module-level handlers
3. Project defaults from `[effects]` in `bock.project`

Layer 3 references the `[effects]` table that Appendix A.3 has marked Reserved for v1.x. The two need to align.

Mark Layer 3 Reserved for v1.x in §10.3:

> **Layer 3 — Project defaults (Reserved for v1.x).** A future `[effects]` table in `bock.project` will allow project-level default handlers. The table is Reserved in Appendix A.3 pending the design pass for project-scoped effect configuration. In v1, effect handler resolution uses Layers 1 and 2 only.

The diagram describing the three-layer resolution is updated to mark Layer 3 with a "(v1.x)" label rather than struck. Keeping Layer 3 visible in the diagram preserves the design intent and gives v1.x a clear target.

### §10.4 Lambda handlers deferred

Current §10.4 illustrates inline lambda-based handlers via `Effect.handler(...)`:

```bock
with Logger as Effect.handler((msg) => print(msg))
```

Implementation accepts the record-shaped form only:

```bock
with Logger as { log: (msg) => print(msg) }
```

The `Effect.handler(...)` form does not parse. D2's example-execution verification surfaced this gap.

Mark the lambda form deferred to v1.x in §10.4:

> **Status:** v1 supports record-shaped handlers (illustrated below). The `Effect.handler(lambda)` lambda-form constructor is planned for v1.x as ergonomic shorthand for single-method effects. The lambda form is preserved here as design intent.

The record-shaped handler examples in §10.4 remain as the v1 surface. Lambda-form examples are caveated.

### §10.6 Effect erasure softened

Current §10.6 describes effect erasure as an optimization where the compiler removes effect tracking from code paths that prove unreachable for a given effect set. The wording is permissive at the surface ("the compiler can inline") but reads as a v1 commitment in context.

The softening adds an explicit "MAY apply" qualifier and a forward reference:

> Effect erasure is a permissive optimization. The compiler MAY apply effect erasure to code paths where the effect set is provably empty after specialization. Implementation of effect erasure is not required for v1 conformance. v1.x may schedule erasure as a dedicated optimization pass; see ROADMAP.md.

The body of §10.6 describing the erasure mechanism remains as guidance for the v1.x optimization work.

## Rationale

**§10.3 inconsistency.** Earlier amendments to Appendix A added the `[effects]` Reserved marker without updating §10.3's prose Layer 3 description. A reader following the spec encounters Layer 3 as a v1 feature in §10.3, then encounters its configuration surface as Reserved in A.3. The two descriptions disagree. Marking Layer 3 explicitly Reserved in §10.3 aligns them.

**§10.4 lambda form.** Record-shaped handlers are more flexible (they support multi-method effects) and currently work. The lambda form is an ergonomic shorthand for the single-method case. The ergonomic gap is real but minor; record-shaped handlers are not awkward enough to block on. Deferring the lambda form to v1.x is the lower-cost path.

**§10.6 overcommitment.** The §10.6 text describes effect erasure with enough specificity that a reader could interpret it as a v1 commitment. Current impl does not implement erasure. Either v1 ships erasure or §10.6 softens; softening is the lower-cost path and preserves the design idea for v1.x. The "MAY apply" wording matches the §17.4 strictness vocabulary (MUST / SHOULD / MAY) and makes the permissive intent unambiguous.

## Implementation Notes

Category: **impl-defined**. Current impl:

- §10.3 Layers 1 and 2 work; Layer 3 is not implemented (consistent with deferred state)
- §10.4 record-shaped handlers work; lambda form does not parse (consistent with deferred state)
- §10.6 effect erasure is not implemented (consistent with softened "MAY apply" wording)

The spec changes align the prose with the existing impl state. No code changes are required.
