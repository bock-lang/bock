## Background

Appendix A's example `bock.project` was aspirational — it
included nine fields the v1 compiler does not read. Documentation
buildout (D1) surfaced this as nine drift rows. Without a rewrite,
new users authoring `bock.project` files would expect fields to
work that don't, and the design chat would keep getting "why
doesn't [field] do anything?" questions.

---

## Structure Change

Appendix A is now three subsections:

- **A.1** — `bock.project` (v1) — the actual fields the v1
  compiler reads
- **A.2** — `bock.package` (v1) — points at §19 for dependency
  declarations (fixes F22)
- **A.3** — Reserved for future versions — fields appearing in
  older drafts that may ship in v1.x or later

This shape prevents future drift: anyone adding a field to the v1
example must know it's actually implemented. Anyone wanting to
preserve a future-looking field puts it in A.3 with a one-line
note on what triggers its activation.

## Field-by-field Resolution

### Removed (no semantic content the compiler acts on)

- **`[paradigm]`** — Bock is multi-paradigm by design; no
  enforcement, no lint surface, no codegen consequence. Not
  "deferred" — there is no design intent to reintroduce.
  If a future lint/style feature needs paradigm declaration,
  it's a different feature with a different name.

### Moved (placement was wrong)

- **`[dependencies]`** — Belongs in `bock.package`, not
  `bock.project`. §19 documents the package manifest;
  Appendix A.1 no longer contains dependencies, and A.2
  shows the correct location with cross-reference to §19.
  Closes F22.

### Reserved (real future work, just not v1)

- **`[project] authors`** — Reserved; activated alongside
  `bock pkg publish` (F07, deferred in 20260512-1400).
- **`[strictness.overrides]`** — Per-path glob-based strictness
  mappings. v1 ships flat project-level strictness; layered
  strictness via globs is v1.x.
- **`[effects]` + `[effects.overrides.<env>]`** — Project-level
  effect handler routing. Marked Reserved with explicit note:
  v1 ships inline + module-level handler resolution (§10);
  project-level routing may be unnecessary and will only be
  introduced if real-world usage demonstrates need. This
  prevents v1.x reading "Reserved" as "we promised this."
- **`[plugins]`** — Plugin declarations. Reserved pending the
  plugin loader; Appendix C describes the planned plugin system.
- **`[testing]`** — Smart target selection thresholds. Reserved
  alongside the deferred cross-target test flags (§20.4, C-1
  in 20260512-1400).
- **`[build.hooks]`** — Pre/post-build script hooks. Reserved
  pending a design pass on hook semantics (ordering, error
  propagation, sandboxing).
- **`[build.cache] remote`** — Remote build cache. Reserved
  pending a v1.x cache-server design.

### Renamed (bundled copy-edit)

- **`[build] min_aura` → `[build] min_bock`** — Predates the
  Aura → Bock rename (2026-04-20). Appendix A wasn't refreshed
  at that time. The field itself remains Reserved (v1 does not
  enforce a minimum compiler version on the manifest); the
  rename brings the example into alignment with the rest of the
  Bock-named spec.

### Kept (v1-active)

- `[project]` — `name`, `version`, `type` (per 20260427-1000)
- `[strictness]` — `default`
- `[targets.<T>]` — deep tooling config (per 20260427-1000)
- `[targets.<T>.scaffolding]` — shallow tooling config
- `[ai]` — full v1 schema unchanged
- `[registries]` — `default` + named registries (per F21
  reclassification; impl reads these)

## Format Decision: Explicit Reserved Section vs. Comments

Two options considered:

- **Option A:** Inline future fields in A.1 with `# RESERVED:`
  comments explaining when they activate.
- **Option B:** Separate A.3 subsection listing reserved fields.

**Decision: Option B.**

Option A would put aspirational content directly in the example
users copy from. Even with comments, users copy-paste examples
faster than they read them, and TOML parsers ignore comments
silently. Option B keeps A.1 as a clean v1 reference; A.3 is a
clearly-labeled "future work" subsection that's hard to mistake
for active configuration.
