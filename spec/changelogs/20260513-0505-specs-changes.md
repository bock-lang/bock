# AIR serialization and binary packages deferred Post-v1

**Date:** 2026-05-13
**Affects:** §16.3, §16.4
**Type:** deferral
**Part of:** D1+D2 spec alignment consolidation

## Summary

The AIR-T (text/JSON) and AIR-B (binary) serialization formats described in §16.3 are deferred Post-v1. Binary package compatibility in §16.4, which depends on §16.3, defers with it. Both are infrastructure for downstream features (build cache reuse, binary distribution) that are not on the v1 critical path.

The Post-v1 label is used in preference to v1.x because there is no immediate roadmap commitment for these surfaces. They are future considerations, not near-term plans.

## Change

### §16.3 AIR serialization formats

Current §16.3 describes AIR-T as a text/JSON serialization and AIR-B as a binary serialization for the Annotated Intermediate Representation. Neither format is implemented in v1.

Add a leading note to §16.3:

> **Status:** Deferred Post-v1. AIR is an internal intermediate representation in v1; serialization is not exposed. The AIR-T and AIR-B formats remain in this section as a planning sketch for future cross-tool interop, build cache reuse, and binary package distribution work. Format details are non-normative.

The §16.3 body text remains as a planning sketch.

### §16.4 Binary package compatibility

Current §16.4 describes how packages distributed in AIR-B form would be compatibility-checked against the consuming compiler version. The mechanism depends entirely on §16.3.

Add a leading note to §16.4:

> **Status:** Deferred Post-v1 with §16.3. v1 distributes packages in source form only.

The §16.4 body text remains as a planning sketch.

## Rationale

AIR is an internal compiler IR. Serialization is useful for three downstream capabilities:

1. Cross-tool interop (third-party tools reading AIR)
2. Build cache reuse (pre-compiled AIR cached between builds)
3. Binary package distribution (packages distributed as pre-compiled AIR)

None of these are on the v1 critical path. Cross-tool interop is an ecosystem capability that develops once the ecosystem exists. Cache reuse is performance polish. Binary distribution is a package-manager enhancement.

Marking these v1.x would imply near-term commitment. They are better described as Post-v1 because there is no current roadmap entry, and committing now would create roadmap pressure without corresponding value.

The §16.3 and §16.4 content is preserved (rather than struck) because the planning sketch is useful for downstream design when the time comes. The leading status note makes the deferral explicit.

## Implementation Notes

Category: **impl-defined**. Current impl matches the deferred-state spec: AIR is in-memory only, packages are distributed in source form.
