## §20.1 — CLI Restructure

**Decision:** Adopt Resolution C with tightened command semantics.
`pin`, `unpin`, `override`, and `cache` become top-level commands.
The spec follows the implementation's empirically-refined surface.

**Rationale:** Verbs describing complete operations should be
top-level commands. `bock pin <id>` is shorter and more
discoverable than `bock override --pin <id>`, and the operations
are conceptually distinct:

- `pin` / `unpin` — pin lifecycle (presence/absence of pin metadata)
- `override --choice=X` — edit which alternative is selected
- `override --promote <id>` — migrate runtime decision to build
- `cache` — operational housekeeping on disk caches

Three orthogonal concerns, three commands. The spec separates
them explicitly so the conceptual structure is visible to anyone
reading §20.1 in isolation.

**Standing principle established:** When CLI ergonomics evolve
through implementation experience, the spec follows. The spec is
normative for capabilities and behavior, not for the precise
shape of the command surface. A new opening paragraph in §20.1
states this explicitly to prevent future "spec vs implementation"
ambiguity.

**Capability gap fixed:** `bock cache` was missing from §20.1
entirely, despite §17.4 (decision manifest), §17.7 (rule cache),
and Appendix B (`.bock/cache/`) referencing the cache subsystem.
§20.1 now acknowledges `bock cache` with subcommands
`list`/`clear`/`prune`/`stats`.

### Section structure

§20.1 reorganized into three thematic groups:
- **Build and execute** — `new`, `build`, `run`, `check`,
  `test`, `fmt`, `fix`, `repl`
- **Decision and rule manifest management** — `inspect`, `pin`,
  `unpin`, `override`, `cache`
- **Project lifecycle** — `promote`, `migrate`, `doc`, `pkg`,
  `model`, `target`, `ci`

This grouping makes the command surface scannable for someone
new to the CLI without changing what commands exist.

## §20.7 — Project Scaffolding (NEW)

**Decision:** Resolution B from the open item — `bock new`
generates a `bock.project` containing a commented-out `[ai]`
block. AI configuration is opt-in. No interactive prompts during
scaffolding.

**Rationale:**
- Resolution A (no `[ai]` block) leaves a discoverability gap.
  Users who want AI features must read external docs to know
  where to configure them.
- Resolution C (interactive prompt) adds a code path that needs
  maintenance and tests, fails awkwardly in CI/scripted contexts,
  and demands provider knowledge from someone whose first
  interaction with Bock is `bock new`.
- Resolution B costs nothing, is self-documenting, and respects
  the deterministic-fallback default.

**Honest framing baked into the comment:** The scaffolded comment
opens with "Bock uses rule-based code generation by default."
This frames AI as opt-in augmentation rather than the primary
codegen path, which matches reality at v1 launch where the AI
path has not yet been characterized at scale.

**Non-prescriptive provider choices:** The commented block uses
placeholder values (`endpoint = "..."`, `model = "..."`) rather
than vendor-specific defaults. The scaffolder does not pretend
to know what provider the user has access to.

### New subsection

§20.7 — Project Scaffolding — added between §20.6 (Build System)
and §21 (Formal Grammar). Specifies the generated file structure,
the commented-`[ai]`-block default, and the explicit decision
against interactive prompts.
