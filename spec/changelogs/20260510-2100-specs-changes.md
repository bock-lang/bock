## Resolution B1 — Bin / Lib Determination

**Decision:** Resolution C with tightened inference rule.

Inference:
- `src/main.bock` with a `main` function → binary
- `src/lib.bock` → library
- Both present → both (where target supports it)
- Neither → error at scaffold time

Override: `[project] type = "bin" | "lib" | "both"` in `bock.project`.

**Why "both" matters:** Rust natively supports `[[bin]]` + `[lib]`
coexistence in a single crate, which is the standard pattern for
CLI tools that also expose programmatic API. Forcing users to
scaffold twice for this case is awkward; supporting "both" as a
first-class kind costs little and matches a real ecosystem
pattern. Targets that don't represent the distinction (Go, JS)
produce a binary and emit a warning that the library surface is
not separately exported. This is recoverable — users targeting
both Rust-as-library and JS-as-binary can use `--target` with
specific output expectations per target.

## Resolution B2 — Per-Target Naming

**Decision:** Resolution C with documented inference rules.

Default normalization rules (committed to spec so users can
predict output):

| Target  | Default rule |
|---------|-------------|
| npm/TS  | Project name preserved as-is (kebab-case is npm-idiomatic) |
| Python  | Hyphens → underscores, leading digits prefixed, special chars stripped (PEP 8) |
| Rust    | Same normalization as Python (binary names cannot contain hyphens) |
| Go      | Placeholder `example.com/<project-name>` with build warning; user must override |

Override sections in `bock.project`:

```toml
[targets.go]
module = "github.com/user/my-project"

[targets.python]
package = "my_custom_name"

[targets.rust]
crate = "my_crate"

[targets.npm]
package = "@scope/my-project"
```

**Go's special treatment:** Pure inference for Go produces
something users almost certainly need to change before pushing
code. The scaffolder emits a warning rather than letting the
placeholder pass silently — this is a foot-gun avoidance pattern.
Users who legitimately want the placeholder for local-only
projects can suppress the warning explicitly.

## Resolution B3 — Lockfile Policy

**Decision:** Resolution B (don't generate). Scaffolded README
includes target-appropriate first-contact instructions.

Scaffolder does not generate `package-lock.json`, `Cargo.lock`,
`poetry.lock`, `go.sum`, or equivalents. Lockfiles are
target-package-manager territory.

Per-target first-contact commands documented in scaffolded README:
- JS: `npm install`
- TS: `npm install && npx tsc`
- Python: `pip install -e .`
- Rust: `cargo build` (handles lockfile)
- Go: `go mod tidy` (handles module graph)

The README addition is the load-bearing part. Without first-contact
instructions, users hit "module not found" errors on initial run
and have to debug the install step.

## Resolution B4 (new) — Test Inclusion in Project Mode

**Decision:** Tests included by default; `--no-tests` opts out.

Project mode includes Bock tests transpiled to the target's
idiomatic test framework. After `bock build --target js`, users
running `npm test` execute the Bock tests as Vitest/Jest tests;
`cargo test` executes them as Rust tests; `pytest` executes them
as Python tests; and so on.

**Rationale:** Bock already commits to semantic equivalence
between source and transpiled output (§17.6 capability gap
resolution; §20.4 Tier 2 transpilation tests). Including tests
in project mode output makes that commitment empirically
checkable in user CI. This is the validation surface that
distinguishes Bock's cross-target semantic equivalence claim
from a "good enough" transpiler — hiding it behind an opt-in
flag understates what the language offers.

**Opt-out cases:** Vendor distribution where tests should not
ship; library consumers wanting only runtime code; security-
sensitive contexts. These are real but minority cases.

**Per-target test framework mapping** is a codegen package
responsibility. The spec doesn't dictate Vitest vs Jest, pytest
vs unittest, etc. — those are empirical choices that should
evolve with the target ecosystems.

**§20.4 amendment.** Tier 2 transpilation tests now serve as the
project mode validation gate. A target's codegen is considered
project-mode-ready when its Tier 2 tests pass on a representative
test suite. Targets where Tier 2 tests fail intermittently or on
common patterns ship source mode (`--source-only`) by default
until the gap closes. This is a release-readiness criterion, not
a user-facing distinction.

## Implementation Implications

This significantly reshapes the Item B implementation plan:

**Phase 1 effort estimate increases from 4-6 hours to 8-12 hours.**
Test transpilation infrastructure was not in the original scope.
It's now Phase 1's responsibility — establishing the foundation
that Phases 2-5 inherit.

**Per-phase acceptance criteria gain a Tier 2 test gate.** Each
target's project mode work (Phases 2-5) must demonstrate Tier 2
transpilation tests passing on a representative test suite before
the phase ships. Without this, project mode for that target ships
untrusted.

**Phase 1 scope additions:**
- `OutputMode` enum and pipeline threading (existing)
- `--source-only` and `--deliverable` flags (existing)
- `--no-tests` flag (new)
- `ProjectMeta` extraction from `bock.project` including new
  optional fields (`[project] type`, `[targets.<T>]`) (existing,
  scope expanded)
- Bin/lib inference logic (new)
- Per-target name normalization logic (new, used by all targets)
- JavaScript scaffolding template — manifest + README + gitignore
  (existing)
- **JavaScript test transpilation** — `@test fn` → Vitest tests,
  test files under `tests/` mirroring source structure, npm test
  script wiring (new, foundational)
- End-to-end test: scaffolded project + transpiled tests pass
  via `npm test` (existing, scope expanded)

**Phases 2-5 inherit test transpilation infrastructure** from
Phase 1. Each per-target phase adds its target's test framework
mapping (pytest, cargo test, go test, jest if not Vitest). This
is bounded work similar in scope to manifest scaffolding, just
applied to test framework conventions instead of project
manifest conventions.

**Phase 6 (default switch) unchanged.** Two-line code change plus
docs/examples migration.

**Calendar duration likely extends to 2-3 weeks elapsed** (from
1-2 weeks in the original plan), driven primarily by Phase 1's
larger scope.
