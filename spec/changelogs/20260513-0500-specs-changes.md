# FFI deferred to v1.x

**Date:** 2026-05-13
**Affects:** §14.1, §14.2
**Type:** deferral
**Part of:** D1+D2 spec alignment consolidation

## Summary

The native FFI surface described in §14.1 (native blocks with backtick tokenization, per-target inline code validation, capability gap interaction) is deferred to v1.x. The dependent FFI linter warning in §14.2 defers with it. The `native` keyword reservation is preserved so that v1.x can introduce the full surface without a breaking lexical change.

## Change

### §14.1 Native blocks

Current §14.1 describes native blocks as a v1 capability. The keyword `native` is tokenized in the lexer today, but no parser handling, validation, or codegen exists. This changelog reframes §14.1 to reflect that state.

Add a leading note to §14.1:

> **Status:** The `native` keyword is reserved (tokenized) in v1. The full native block surface (parsing, per-target inline code validation, capability gap interaction) is planned for v1.x. v1 code that uses `native` blocks will fail at parse time with a "feature reserved for v1.x" diagnostic. The keyword reservation prevents v1.x from being a breaking lexical change.

The §14.1 body text describing native block semantics remains in place as the planned v1.x surface. It is informative for users tracking the roadmap, not normative for v1.

### §14.2 FFI linter warning

Current §14.2 describes a linter warning that fires when native blocks are used without the appropriate capability declarations. The warning is meaningless until §14.1 parses native blocks.

Add a leading note to §14.2:

> **Status:** Deferred to v1.x with §14.1. The warning becomes active when native blocks parse.

The §14.2 body text remains as the planned v1.x surface.

## Rationale

FFI is a discrete capability that warrants its own design pass when the time comes to ship it. The pass needs to resolve several questions:

- Backtick tokenization edge cases (nested backticks, escaping, multiline)
- Per-target inline code validation (target-specific syntax checking)
- Interaction with the capability gap synthesis described in §17.6
- Interaction with §15 conditional compilation annotations (`@target`, `@platform`)

Forcing a half-built FFI surface into v1 to match the spec would either underdeliver on these design questions or block v1 on FFI completeness. Deferring the surface as a unit and reserving the keyword is the lower-risk path. v1 ships without native FFI; users requiring native interop wait for v1.x.

## Implementation Notes

Category: **impl-defined**. Current impl matches the deferred-state spec: `native` keyword tokenizes, no further handling exists. The diagnostic suggested above ("feature reserved for v1.x") can land as a small impl follow-up but is not blocking; the existing parse error is acceptable in the interim.
