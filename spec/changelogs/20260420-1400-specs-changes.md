# Spec Changes — 2026-04-22 14:00

Two near-term questions addressed following Phase D readiness pass:
cancellation model for adaptive handlers, and decision manifest
structure for runtime vs build-time decisions.

---

## Decision 1: Cancellation Model

**Decision:** Cancellation is an ambient effect (`Cancel`) in all
async contexts. Cooperative, checkpoint-based, with targets
mapping to their native mechanisms (tokio CancellationToken,
AbortSignal, context.Context, asyncio cancellation).

**Rationale:** The spec had no cancellation semantics at all —
a gap that would have blocked both adaptive handler implementation
and general async correctness. Three approaches were considered:
ambient effect (Go/tokio-style cooperative), exception-like
(Python/C#-style throw), and capability-based (novel). Ambient
effect was chosen because:

1. Aura already has an effect system with ambient effects
   (Panic, Allocate) — cancellation fits naturally
2. Transpilation is clean: each target has a well-understood
   cancellation primitive
3. No exception-based control flow (Aura has avoided exceptions
   elsewhere)
4. Checkpoints map naturally to effect operation invocations
   — places where cancellation matters most

### §13.5 — Cancellation — NEW
New subsection under §13 Concurrency. Covers:
- Ambient `Cancel` effect available in all async contexts
- Checkpoint semantics (await boundaries, effect operations,
  `check_cancel()`, loop iterations in `@concurrent`)
- Task handle with `cancel()` method
- Structured concurrency: parent cancellation propagates to children
- `Cancelled` as typed return value (not exception)
- `?` operator propagates Cancelled like Err
- Target mapping table (Rust/JS/Go/Python)
- Strictness interaction: sketch auto-inserts, development warns,
  production errors on missing checkpoints beyond depth bound

### Adaptive Handlers: Cancellation Integration
§10.8 `RecoveryStrategy` trait extended:
- `attempt` return type: `Result[T, E] | Cancelled`
- New `on_cancel(context)` hook with default no-op
- Strategy cleanup semantics specified
- Built-in combinators respect cancellation at await points

## Decision 2: Decision Manifest Split

**Decision:** Split manifest storage into `build/` and `runtime/`
subdirectories. Build decisions committed to VCS; runtime
decisions environment-local. Unified UI via `aura inspect` with
scoping flags.

**Rationale:** Build-time codegen decisions and runtime adaptive
handler selections have fundamentally different lifecycles and
volumes. Build: dozens to hundreds per project, stable artifacts.
Runtime: potentially thousands per day in production, time-bounded
telemetry. Mixing them either drowns build signal in runtime
volume or requires filtering that exposes the split anyway.

Separation at storage, unification at UI preserves the single
mental model while acknowledging the structural difference. The
promotion path (`--promote`) gives developers an explicit way to
move from AI-selected runtime behavior to build-committed
deterministic behavior — a critical transition for production
readiness.

### §17.4 — Decision Manifest — EXPANDED
Added "Build-time vs runtime decisions" paragraph distinguishing
the two populations. Storage locations specified:
- `.aura/decisions/build/` — codegen, committed to VCS
- `.aura/decisions/runtime/` — adaptive handlers, local only

CLI behavior specified:
- `aura inspect` defaults to build
- `aura inspect --runtime` for runtime
- `aura inspect --all` for unified view
- `aura override --runtime` for runtime pins
- `aura override --promote <id>` for runtime → build promotion

### §10.8 — Runtime Decisions — EXPANDED
New "Runtime decisions in the manifest" paragraph referencing
the split. New "Promotion path" paragraph describing how
stabilized runtime selections migrate to build-time pins.

## Implementation Implications

**Phase D (AI provider + adaptive handlers):** Can now implement
`RecoveryStrategy` with the full signature including `on_cancel`.
Decision logging should write to `.aura/decisions/runtime/` from
the start. The `aura override --promote` command is a Phase 7
CLI concern but should be planned for.

**Phase 5/6 (interpreter + core runtime):** Will need to implement
the cancellation primitive — `CancellationToken` equivalent in
the interpreter's async runtime, checkpoint insertion in the AIR
lowering, `check_cancel()` as a prelude function, task handle with
`cancel()` method.

**Phase 9 (codegen):** Each target generator must map the `Cancel`
effect to its native primitive per the table in §13.5.

No changes to the 61-package plan. Cancellation implementation
threads through existing packages (P5.5 effect handler runtime,
P6.x core library additions, P9.x target generators) rather than
requiring new packages.
