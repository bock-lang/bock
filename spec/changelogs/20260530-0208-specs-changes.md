# Stdlib design batch — prelude `Ordering`, sealed primitive conformances, per-package strictness, core.error v1 surface, v1 import forms

**Date:** 2026-05-30
**Affects:** §1.4, §12.2, §18.2, §18.3 (`core.error`), §18.5; supersedes the `core.error` v1 surface recorded in `20260529-2251-specs-changes.md`; `stdlib/CLAUDE.md` (non-spec, framing)
**Type:** clarification + addition (Design-chat stdlib-batch decisions, 2026-05-30; reconciliation/prose only — no checker or runtime behavior changes in this change set)

## Change

The 2026-05-30 Design-chat stdlib-batch decisions (Q1b, Q1c, Q1d, Q2,
Q3, Q4) are landed as spec prose and `stdlib/CLAUDE.md` framing. These
are documentation/normative-prose changes only; the implementation work
(the primitive-conformance bridge, prelude injection, import
diagnostics) lands separately and is **not** part of this change set.

### Q4 — §18.2 prelude: `Ordering` + variants added; prelude model clarified

`Ordering`, `Less`, `Equal`, and `Greater` are added to the §18.2
"always available without import" list, placed alongside
`Optional`/`Some`/`None` and `Result`/`Ok`/`Err`. Rationale: `Comparable`
is a prelude trait and its `compare` method returns `Ordering`, so
`Ordering` and its variants must be nameable without an import to write
a `compare` implementation or to `match` the result of a `cmp()` call.

§18.2 also gains a clarifying statement that the prelude is the
authoritative **auto-imported subset, re-exported from the `core.*`
modules where the symbols are defined.** §18.2 (what is auto-imported)
and §18.3 (where each symbol is defined) are thereby explicitly
consistent rather than contradictory: a prelude name is the
prelude-visible re-export of a definition that lives in some `core.*`
module.

### Q1b — §18.5 sealed primitive conformances (coherence)

A normative paragraph is added to §18.5: the canonical trait
conformances for **primitive types** (e.g. `Int: Comparable`,
`Int: Equatable`, `String: Displayable`, `Bool: Equatable`) are provided
by `core` and are **sealed.** User code may **not** implement a core
trait for a primitive type; the compiler rejects such an `impl` as a
coherence (orphan-rule) violation. A user needing different behavior
wraps the primitive in a newtype (e.g.
`record Descending { value: Int }` with its own `impl Comparable for
Descending`). The paragraph is explicitly scoped: it pins down only the
`(core trait, primitive type)` quadrant; the broader orphan/coherence
model for `(user trait, user type)`, `(core trait, user type)`, and
`(user trait, primitive type)` is **not specified here.** No error code
is invented — the spec states the rule; the diagnostic number is the
implementation's call.

### Q1c — distribution mechanism stays non-normative

No embedded-source/shims mechanism is added to the normative §18 body.
§18.1's contract (`core` ships with the compiler, small, stable, every
target) is already the normative part and is unchanged. `stdlib/CLAUDE.md`
is updated so its `core/` section frames the embedded-source-compiled
loading mechanism, the per-target shim layout, and the `$BOCK_STDLIB`
override as an **implementation strategy** that may evolve (e.g. toward
pre-compiled AIR per §16.4) without a spec change, with the **normative
contract being §18.1.** The stale per-target `bock-codegen-<target>`
shim path is corrected to the actual single-crate
`compiler/crates/bock-codegen/` layout.

### Q1d — §1.4 per-package strictness

A normative paragraph is added to §1.4: **strictness is a property of a
compilation unit (package), declared by that package's author.** The
`bock.project` strictness governs the user's own package. Dependencies —
the bundled `core` stdlib today, third-party packages once the package
manager ships (§19) — compile at **their own** declared strictness; a
dependency that compiles successfully is trusted, and its non-error
diagnostics are not surfaced as the consumer's build errors. A
consumer's strict (`production`) build never fails because of a
dependency's internals. The level the bundled stdlib itself compiles at
is deliberately **not** specified normatively — that is a non-normative
quality bar (STATUS/ROADMAP), not the contract.

### Q2 — §18.3 `core.error` v1 surface = `message()`-only

The §18.3 `core.error` line keeps its v1 marker and appends the
v1-minimal scope: the v1 surface is **`message(self) -> String` only.**
Error chaining (`cause`/`source`), an `Error: Displayable` supertrait,
and context/wrapping helpers depend on **trait objects** (`Error` used
as a type, not as a bound — Reserved for v1.x) and ship together as a
**v1.x error-ergonomics bundle.**

**Supersession (latest decision wins).** This explicitly supersedes the
`core.error` v1 surface recorded in `20260529-2251-specs-changes.md`,
whose Q1 minimum-useful list leaned `source` (→ `Optional[Error]`) into
the v1 surface. That earlier entry is corrected in place (with a dated
supersession note) to `message()`-only, with `source`/`cause` deferred
to the v1.x error-ergonomics bundle on the trait-object dependency.

### Q3 — §12.2 v1 import forms; bare module import rejected

§12.2 already lists the braced form and the wildcard (`*`, discouraged)
as the two v1 import forms and notes aliasing is v1.x; those forms are
confirmed and left intact. A brief clarification is added: a `use` of a
module path with **neither a brace-list nor a wildcard** (a bare
`use core.error`) is **not a v1 import form** — the type checker rejects
it and points at the braced form. Module-qualified access (referring to
symbols as `core.error.Error` rather than importing them) is deferred to
v1.x alongside aliasing. The diagnostic itself is the implementation's
call.

## Rationale

These are the 2026-05-30 Design-chat stdlib-batch decisions, landed
faithfully as prose. Q4 makes `Comparable`'s return type nameable
without an import (a latent gap, since `Comparable` is prelude but
`Ordering` was not) and removes an apparent §18.2/§18.3 contradiction by
naming the re-export relationship. Q1b gives `core` coherence authority
over primitive conformances so the prelude trait set is stable and
unambiguous, while deliberately leaving the general orphan-rule model
open for a later decision. Q1c keeps the distribution mechanism free to
evolve (notably toward §16.4 pre-compiled AIR) by keeping it out of the
normative §18 body. Q1d makes strict builds composable: a consumer's
strictness governs only the consumer's code, so adding a dependency
cannot break a strict build on the dependency's internals. Q2 trims the
v1 `core.error` surface to what does not require trait objects, deferring
the rest as a coherent v1.x bundle — and resolves the divergence the
2251 entry's broader lean created. Q3 confirms the two v1 import forms
and closes the bare-module gap so v1 import behavior is unambiguous.

## Migration

- **`core.error`:** v1 `Error` exposes `message()` only. Code relying on
  `source`/`cause`, an `Error: Displayable` supertrait, or
  context/wrapping helpers targets the v1.x error-ergonomics bundle and
  does not compile against v1.
- **Imports:** bare `use core.error` (no brace-list, no wildcard) does
  not compile in v1; use the braced form (`use core.error.{Error}`).
  Module-qualified access (`core.error.Error`) and aliasing are v1.x.
- **Primitive conformances:** an `impl` of a core trait for a primitive
  type (e.g. `impl Comparable for Int`) is rejected as a coherence
  violation; wrap the primitive in a newtype and implement the trait on
  the wrapper instead.
- **Strictness:** raising a package to `production` tightens rules on
  that package's own code only; a dependency's diagnostics are not
  re-surfaced.
