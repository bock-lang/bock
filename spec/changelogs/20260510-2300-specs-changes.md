# Spec Changes — 2026-04-27 10:00 UTC

OPEN B5 resolved: project mode supports user configuration of
per-target tooling via a deep/shallow split. Formatter selection
treated as deep config (affects codegen) after design discussion
revealed that shallow treatment would produce version-control

OPEN B6 (template overrides) confirmed deferred to post-v1.

---

## Background

Item B implementation plan review surfaced an adoption-blocker
question: project mode locks teams into Bock's choice of test
framework, formatter, linter, and package manager. A team
standardized on Jest cannot adopt Bock without post-processing
the Vitest tests Bock would emit. A team using Black gets
codegen that may not match Black's formatting and produces
first-commit churn when Black is applied.

Three resolution shapes were considered:
- A: codegen-only (status quo)
- B: full configuration matrix
- C: hybrid (deep vs shallow categories)

Initial design read favored C with formatter as shallow config.
Design discussion identified that formatters with opinionated
defaults (Prettier, Black, gofmt, rustfmt) dictate code shape;
shipping a formatter config that disagrees with generated code
produces churn rather than just an extra file. Formatter
selection moved to deep config.

---

## Resolution B5

**Decision:** Resolution C with formatter in the deep category.

Two configuration sections per target in `bock.project`:

**`[targets.<T>]` — deep configuration** (changes what code Bock
emits):
- `test_framework` — different framework requires different test
  file structure and assertion APIs (codegen branch required)
- `formatter` — different formatter requires different
  formatting conventions in emitted code (codegen branch required)

**`[targets.<T>.scaffolding]` — shallow configuration** (adds
files alongside emitted code):
- `linter` — config file added; emitted code doesn't change
- `package_manager` — affects scaffolded README first-contact
  instructions only

### v1-supported deep configuration matrix

| Target | Test framework | Formatter |
|--------|---------------|-----------|
| JS     | Vitest (default), Jest | Prettier (default), none |
| TS     | Vitest (default), Jest | Prettier (default), none |
| Python | pytest (default), unittest | Black (default), Ruff format, none |
| Rust   | cargo test (universal) | rustfmt (universal — always on) |
| Go     | stdlib testing (universal) | gofmt (universal — always on) |

Rust and Go have universal, always-on formatters because the
ecosystem has already made the choice. There is no user-facing
formatter selection for these targets; Bock's codegen must
produce formatter-clean output as a release-readiness baseline.

Variant runners that wrap the default framework (Rust's
`nextest`, Go's `testify`) do not require codegen branches —
they read the same test files. Scaffolded READMEs mention them
as options without separate scaffold support.

### v1-supported shallow configuration matrix

| Target | Linter | Package manager |
|--------|--------|----------------|
| JS     | ESLint | npm (default), pnpm, yarn |
| TS     | ESLint | npm (default), pnpm, yarn |
| Python | Ruff check, Pylint | pip (default), Poetry, uv |
| Rust   | Clippy | (cargo only) |
| Go     | golangci-lint | (go mod only) |

Shallow tooling is opt-in. Omitting the field generates no
config file. Selecting generates a baseline config that doesn't
conflict with Bock's emitted patterns.

## Why Formatter Selection Is Deep, Not Shallow

The original design read placed formatter in the shallow
category alongside linter. Design discussion identified this as
incorrect.

A formatter with opinionated defaults (Prettier, Black, gofmt,
rustfmt) dictates code shape. If Bock emits JS with tabs,
double-quoted strings, and trailing semicolons, then ships a
`.prettierrc` defaulting to `"useTabs": false, "singleQuote":
true, "semi": false`, the user's first action is `prettier
--write .`. Every generated file gets rewritten. The first
commit on the project is enormous and obscures any later real
changes.

The principle: **a formatter config that disagrees with the
generated code's style produces churn, not just an extra file.**
Including the config without making codegen respect it is worse
than not including the config at all.

Therefore: when a formatter is configured (or universal as in
Rust/Go), Bock's emitted code must pass the formatter's check
cleanly on first generation. This is the "codegen-formatter
agreement" and it's enforced as a release-readiness criterion
via the §20.4 validation gate.

Linters differ. ESLint, Clippy, golangci-lint flag issues but
don't typically rewrite code (when not used with `--fix` for
formatting rules). Bock generates code that passes the
recommended baseline (eslint:recommended, clippy::all default).
User-specific lint rules are the user's responsibility — if
they conflict, the user either adjusts their config or
post-processes.

## OPEN B6 — Template Overrides

Confirmed: deferred to post-v1.

User-supplied template overrides for `package.json`,
`tsconfig.json`, `pyproject.toml`, etc. (via a hypothetical
`[scaffolding] templates_dir` field) are a real future need but
out of scope for v1.

The v1 scope establishes *configurable categories* (test
framework, formatter, linter, package manager). v1.x or post-v1
work can layer *template-level customization* on top once the
v1 scope is validated by real usage.

Recording B6 as a known future direction so it isn't lost when
v1 stabilizes.
