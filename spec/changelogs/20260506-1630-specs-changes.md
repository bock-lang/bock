# Spec Changes — 2026-04-26 16:30 UTC

Build output structure formalized as §20.6.1 (output layout)
and §20.6.2 (output modes). §20.1 updated to reflect new flags
on `bock build`. s13-transpilation excerpt updated with mode
section and tier/mode disambiguation.

---

## Background

Two related concerns surfaced from the implementation chat:

1. **Output filename convention.** `/get-started` page copy
   showed `node build/js/main.js` while CLI emitted
   `build/js/output.js`. Question: should filenames mirror
   source, and should the spec formalize it?

2. **Output completeness.** Current implementation produces
   bare source-language output with no scaffolding. The spec
   implied complete output (§17.5 deliverables) without
   defining the structural commitment between bare source and
   final deliverable. Implementation needs clear boundaries
   on what `bock build` produces by default.

Both resolved in this changelog as related amendments to §20.6.

---

## §20.6.1 — Output Layout (NEW)

**Decision:** Build output mirrors source filesystem structure.
A source file at `src/<path>.bock` produces output at
`build/<target>/<path>.<ext>`. Module nesting is preserved.

**Rationale:** `output.js` worked for exactly one source file in
one target. Multi-file projects collide or require ad-hoc
disambiguation. Source mirroring matches every other transpiler
in the targets' ecosystems (tsc, babel, swc, rustc), scales to
arbitrary project structure, and preserves the module hierarchy
that §12 already treats as semantically significant.

**Section placement:** §20.6 (Build System), not §13
(Transpilation). The mental separation: §13 is about
transformation rules ("ADTs in Go become tagged structs"); §20.6
is about where the resulting bytes land. Filename conventions
are a build concern, not a transformation concern.

Three paragraphs added:
1. Source-mirror principle with per-target extension
2. Acknowledgment that target-specific scaffolding files live
   alongside the mirrored source structure
3. Entry-point selection is a project-level concern
   (`bock.project`); default is `src/main.bock` when present

**Standing principle (carried from 20260506):** Output filenames
are part of the user-facing consumption surface in a way that
CLI flags are not. Filenames get embedded in CI scripts, deploy
pipelines, import statements, run commands. They are behavior;
they belong in the spec.

## §20.6.2 — Output Modes (NEW)

**Decision:** `bock build --target T` produces output in one of
three modes selected by flag. Default is project mode (scaffolded
target-language project, runnable in the target's normal
toolchain).

| Mode | Flag | Output |
|------|------|--------|
| Source | `--source-only` | Source files only, no scaffolding |
| Project | (default) | Source files + target-ecosystem scaffolding |
| Deliverable | `--deliverable` | Final runnable artifact |

**Default rationale:** `bock build` implies producing something
the user can run. Source mode exists for the
integration-into-existing-project case. Deliverable mode exists
for production deployment and aligns with §17.5.

**Terminology decision (avoiding clash with §17.2).** §17.2
already uses "Tier 1/2/3" for AI involvement axis (AI
generation / rule-based / AI optimization). The output
completeness model uses a *different axis* and would create
genuine confusion if it shared "tier" labels. The new section
explicitly uses "modes" rather than "tiers" and notes the
distinction at the top of §20.6.2. The s13-transpilation excerpt
likewise distinguishes "Three-Tier Generation (AI involvement)"
from "Output Modes (output completeness)" with explicit headings
and section references.

**Spec depth principle:** The spec commits to the *mode model
and structural commitment* (what each mode produces, which flag
selects it, what the default is). Per-target scaffolding details
(`package.json` field contents, `Cargo.toml` layout, Python
project structure) are documented in each target's codegen
package. This gives implementation clear boundaries without the
spec growing into a per-target manifest enumeration that would
lock in choices that should be empirical.

**§17.5 unchanged.** §17.5 already describes deliverable types
(APK, IPA, Docker, deployment packages) and target manifest
configuration. §20.6.2's deliverable mode references §17.5 for
detail. No restructure needed — §17.5 was already deliverable-
focused; the gap was the missing project-mode middle layer,
which §20.6.2 now provides.

## §20.1 — Updated `bock build` Flags

`bock build` description updated to reference output modes and
the new flag list:

```
bock build — Transpile and compile. Produces a scaffolded project
(project mode) by default; see §20.6.2 for output modes. Flags:
--target, --all-targets, --source-only, --deliverable,
--deterministic, --optimize, --release.
```

Added: `--source-only` for source mode. `--deliverable` was
already present and remains. Order tightened to put mode flags
together.

## Implementation Implications

**Today: source mode only.** Current `bock-codegen` produces
bare source files. This is now formally "source mode" — a
supported configuration accessible via `--source-only`, not a
deficiency relative to spec.

**Project mode is the next codegen extension.** Per-target
scaffolding work — `package.json` generation for JS/TS, `Cargo.toml`
for Rust, `pyproject.toml` for Python, `go.mod` for Go. Each
target's codegen package owns its scaffolding template. This is
substantial work but bounded: one set of templates per target.
The spec amendment establishes the scope; codegen packages
implement.

**Deliverable mode is post-v1.** Crosses into containerization,
bundling, and platform packaging that uses external tools. §17.5
already lists the deliverable types. Implementation scheduled
post-v1 once project mode is solid.

**Filename derivation work** — already noted in the immediate
implementation queue. `bock-codegen` derives output filename from
source path within `src/`, places at corresponding path under
`build/<target>/`. This applies regardless of output mode.

**Default behavior changes when project mode lands.** Today,
`bock build --target T` produces source-only output (informally,
because that's all that's implemented). When project mode lands
as default, `bock build --target T` will produce a scaffolded
project. Users who want today's behavior switch to
`--source-only`. Pre-1.0 default change is acceptable;
implementation chat should note this in release communications
when project mode ships.
