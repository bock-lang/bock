## Spec Amendments (aura-spec.md)

### §5.1 — Ownership: Divergence-Aware Branch Merging
**What changed:** Added paragraph after the core ownership rules
explaining how the compiler handles ownership at control flow join
points.

**Why:** The spec defined ownership rules (move, borrow, mut) but
didn't specify what happens at if/else and match join points,
particularly when one branch diverges. This is critical for the
`guard` pattern and early-return idioms to work without false
use-after-move warnings.

**Rule added:** Diverging branches (`return`, `break`, `continue`,
`Never`-typed) are excluded from ownership state merging at join
points. Non-diverging branches merge conservatively (any move →
moved). Moving inside a loop body is an error.

### §11.8 — Context Composition: Type-Level PII Propagation
**What changed:** Expanded from one sentence to a full paragraph
describing the propagation model.

**Why:** The original text ("Security classifications propagate
across module boundaries") was correct but lacked specificity on
mechanism. The implementation chat had scoped this to direct usage
only (no cross-module propagation), which undercut the spec's
intent. This amendment restores full cross-module propagation while
keeping the analysis tractable.

**Model specified:** Type-level transitive analysis. PII-tainted
type set computed via transitive closure over fields and generic
parameters. Function signatures checked at module boundaries.
Cross-module imports checked. Log/print leak detection. Explicitly
NOT value-level taint tracking.

### §16.2 — AIR Node Structure: Typed Children
**What changed:** Removed `children: List[AIRNode]` from the
AIRNode struct. Added note that children are structurally typed
within each `NodeKind` variant.

**Why:** A flat child list provides no structural guarantees.
Typed children (e.g., `FnDecl { params, body, return_type }`)
mirror the AST design and give the compiler exhaustive matching
at each AIR layer. This was identified during the P3.1 package
review and is the standard approach for typed IR representations.

### §18.3 — core.string: String len() Semantics
**What changed:** Added clarification that `String.len()` counts
Unicode scalar values (characters), not bytes. Added `byte_len()`
for byte count.

**Why:** The spec used `len(self)` in refinement predicates and
the Collection trait but didn't specify whether it counted chars
or bytes. For a high-level language with UTF-8 strings, character
count is the expected default. This was identified during the P6.6
package review.
