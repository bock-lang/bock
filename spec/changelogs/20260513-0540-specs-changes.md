# Refinement types Post-v1; tuple positional indexing v1.x

**Date:** 2026-05-13
**Affects:** §4.7, §7.6
**Type:** deferral
**Part of:** D1+D2 spec alignment consolidation

## Summary

Two type-system extensions illustrated in the spec but not implemented:

- **§4.7 refinement types** with predicate clauses (`type Email = String where (s) => s.contains("@")`). Reserved syntax in the lexer; predicate clause does not parse. **Deferred Post-v1.**
- **§7.6 tuple positional indexing** (`pair.0`, `pair.1`). Tuple destructuring works; positional indexing does not parse. **Deferred to v1.x.**

The two have different dispositions because they have different design weight. Refinement types open substantive design surface (predicate evaluation timing, type compatibility under refinement, target codegen costs); tuple positional indexing is an ergonomic gap covered by destructuring.

## Change

### §4.7 refinement types Post-v1

Current §4.7 illustrates refinement types with worked examples:

```bock
type Email = String where (s) => s.contains("@")
type Positive = Int where (n) => n > 0
```

The `type X = Y` form works as a basic type alias. The `where (...) => ...` predicate clause does not parse. The `where` keyword is tokenized but no refinement parser exists.

Mark §4.7 Reserved Post-v1 and remove the worked examples from spec body:

> **Reserved Post-v1.** Refinement types with predicate clauses are a planned future extension to the type system. The design questions are non-trivial:
>
> - Predicate evaluation timing (construction-only, every assignment, every use?)
> - Type compatibility under refinement (is `Email` assignable to `String`? to another `Email`?)
> - Target codegen costs (runtime predicate checking on every assignment is expensive on hot paths)
> - Interaction with AI capability gap synthesis (predicate-aware synthesis is a research direction)
>
> The worked examples and predicate-clause syntax are removed from v1 spec body. They are retained in design notes for future development.

The §4.7 body retains the basic type alias surface (`type X = Y` without `where`) which works today.

### §7.6 tuple positional indexing v1.x

Current §7.6 illustrates:

```bock
let pair = (42, "hello")
let (a, b) = pair       // destructuring (works)
let first = pair.0      // positional indexing (does not parse)
```

The destructuring form works in v1. Positional `.0`/`.1` access does not parse.

Mark positional indexing deferred to v1.x:

> **Deferred to v1.x: positional indexing.** Tuple positional access (`pair.0`, `pair.1`) is planned for v1.x. v1 supports tuple destructuring (`let (a, b) = pair`), which covers the common case. Positional indexing is ergonomic for cases where you need a single element without destructuring the whole tuple.

The destructuring examples in §7.6 are unchanged. Positional-indexing examples are caveated.

## Rationale

**§4.7 refinement types Post-v1 (not v1.x).** Refinement types are a substantial type-system extension with non-trivial design questions. Marking them v1.x would imply near-term commitment; the design pass to resolve evaluation timing, type compatibility, codegen costs, and AI synthesis interaction has not happened. Post-v1 (no immediate commitment) is the honest disposition.

Removing the worked examples from spec body avoids the trap where readers see `where`-clause examples and assume they work. The Reserved marker preserves the design intent for future development without overpromising.

**§7.6 tuple indexing v1.x.** This is a small ergonomic gap with clear semantics. Destructuring covers most use cases; indexing is the convenience form. Deferring to v1.x is straightforward; the implementation cost is bounded (parser extension + AIR/codegen for the access).

## Implementation Notes

Category: **impl-defined** for both.

- §4.7: refinement type predicate clauses do not parse; `where` is tokenized but no further handling. Spec change matches impl.
- §7.6: positional indexing does not parse. Destructuring works. Spec change matches impl.

No code changes required. A small follow-up impl task: improve the diagnostic when users attempt `tuple.0` syntax in v1 to suggest destructuring. Polish item, not blocking.
