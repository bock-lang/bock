## Background

The 16 section excerpt files (`s01-lexical.md` through
`s16-tooling.md`) served a real purpose during the agentic-
implementation phase: bite-sized spec excerpts that fit in an
agent's context window when implementing specific packages. That
purpose has been served.

Persistent friction from the dual-source structure:

1. **Section numbering mismatch.** Section files were numbered
   `s01–s16`; `bock-spec.md` has §1–§23. The natural assumption
   "`s13-transpilation.md` covers §13" is false (it covers §17).
   Documentation buildout (D2-D5) would have to carry a mapping
   table indefinitely.

2. **Double-update tax.** Recent design-pass changelogs
   (20260426-1630, 20260426-2100, 20260427-1000) touched both
   `bock-spec.md` and the corresponding section files in every
   case. That double-update has been pure overhead — same
   content, two locations, perpetually at risk of drift.

3. **Out-of-context content.** Section files are excerpts written
   for agentic context loading. They duplicate content from the
   spec in less authoritative form. When the spec amendment
   process inevitably forgets one file or the other, the section
   files end up either stale or contradictory.

4. **Authority confusion.** The spec is the single source of
   truth (per the Spec Authority Note). Section files acting as
   a parallel surface dilute that claim every time a reader has
   to ask "but does the spec excerpt agree?"

---

## Decision

**Consolidate now.** Delete the section files. `bock-spec.md` is
the sole authoritative spec document.

Three options were considered:

1. Rename section files to match `bock-spec.md` numbering
2. Document the mapping; don't rename
3. Consolidate and deprecate (chosen)

Option 1 creates link-rot in 18 historical changelogs. Option 2
perpetuates the friction indefinitely with a mapping table. Option
3 removes the friction permanently. The triggering condition for
Option 3 — "the agentic-implementation phase has matured enough
that section files are no longer load-bearing" — is now met.

## Implications for Package References

Work packages currently list spec references in two forms:

- Section file references: `specs/s01-lexical.md`, `specs/s13-transpilation.md`
- Spec section references: `§3.7`, `§11.8`, `§17.6`

After consolidation, only the latter form is valid. Packages need
their spec references converted from file-paths to section-numbers
where they currently use the file-path form.

**This is the primary follow-up work for the implementation chat.**
A separate handoff document (`20260512-1700-k04-handoff.md`)
provides the conversion mapping and the recommended approach.

## Implications for Project Knowledge

The Claude project's uploaded knowledge files (per
`PROJECT-SETUP.md`) currently include the 16 section excerpts.
Project knowledge needs the section excerpts removed and only
`bock-spec.md` retained for spec content.

Token budget impact: roughly neutral. The 16 excerpts totaled
~24-30k tokens; `bock-spec.md` is ~55-60k tokens. The previous
setup may have actually been *more* token-efficient for narrow
queries because excerpts could match without pulling the full
spec. The simplification benefit (single source of truth, no
drift risk, no mapping table) outweighs the small token cost,
especially given that selective-section reading still works on
`bock-spec.md` via the section-number references in packages.
