# Spec Changes — 2026-04-20 11:00

Phase D readiness pass: seven clarifications before implementing
§17.8 (AI provider) and §10.8 (adaptive effect handlers).

Five spec amendments, two items confirmed as correct-as-written.

---

## Context

Phase D represents ~40 hours of implementation on the core
differentiator — AI provider interface plus adaptive effect
handlers. Before locking in implementation decisions, seven
ambiguities were identified and resolved against the spec.

---

## Q1. Provider `select()` Mode — AMENDED

**Decision:** Add Select as a fourth interaction mode on the
provider interface.

**Rationale:** The existing three modes (Generate, Repair,
Optimize) all return code. Strategy selection returns a choice
from a closed set — a structurally different operation. Making
it a distinct mode enforces the closed-set constraint at the
trait level rather than relying on every caller to validate.

**§17.8 amended:** "Interaction modes" expanded from three to
four. Select mode defined: receives closed option set + context,
returns choice identifier + confidence + reasoning. Trait return
type enforces that returned identifier is from the provided set.

---

## Q2. Confidence Type — AMENDED

**Decision:** Float `0.0`–`1.0`, with default acceptance threshold
`0.75`. Configurable via `[ai]`.

**Rationale:** §17.4 example already showed `"confidence": 0.92`.
Pinning the type prevents implementation drift. Threshold default
needed so Phase D can code the accept/fallback boundary.

**§17.4 amended:** Added `confidence` field to example JSON.
Added paragraph specifying float range, acceptance threshold
default, and pin-bypass behavior (pinned decisions skip the
confidence check).

**Appendix A amended:** Added `confidence_threshold = 0.75` to
`[ai]` section.

---

## Q3. AI Invocation Scope — AMENDED

**Decision:** Selective invocation based on capability gaps and
target-flagged constructs. Not every AIR node.

**Rationale:** Invoking AI on every node makes every compile
dependent on AI availability, violates the "deterministic
fallback" guarantee, and is 10x+ more expensive for minimal
benefit. The capability gap table (§17.6) already identifies
which constructs need AI; trivial constructs (literals,
arithmetic, direct calls) go through rules.

**§17.2 amended:** Tier 1 description tightened. "AI generation
is invoked selectively at capability gap points identified by
§17.6 and at AIR constructs the target profile flags as
requiring idiomatic synthesis." Tier 2 reframed: "Handles the
common case by default and serves as the fallback when Tier 1
fails or is unavailable."

---

## Q4. Rule Cache Location — AMENDED

**Decision:** Add `.aura/rules/` to project structure. Leave
internal format as implementation concern.

**Rationale:** Location needs to be specced for cross-project
compatibility (rule packages, shared caches). Format is still
TBD per §17.7 and should remain flexible.

**Appendix B amended:** Added `.aura/rules/` directory with note
that per-target subdirectories contain extracted codegen rules.

---

## Q5. RecoveryContext Contents — AMENDED

**Decision:** Define the struct explicitly. Include error,
operation, annotations snapshot, elapsed, attempt, bounded
history (10 items). Exclude AIR, call stack, source, concurrent
state.

**Rationale:** §10.8 said "receives the error... and annotations"
without pinning what else is available. Leaving this to
implementation risks inflating prompt tokens (full AIR is
expensive) or leaking source structure to AI providers in
violation of `@security` classifications.

**§10.8 amended:** Added "Recovery context" subsection with
complete struct definition in Aura syntax. Explicit exclusion
list explains what is NOT passed and why.

---

## Q6. Adaptive Pinning Granularity — AMENDED

**Decision:** Pin by `(error_signature, operation)` pairs.
`error_signature` = error type + hash of structural properties.

**Rationale:** Matches developer mental model ("connection
timeout on fetch_payment → retry"). Per-decision pinning (Option
B) explodes the manifest. Per-annotation pinning (Option C) is
too permissive.

**§10.8 amended:** Added "Pinning granularity" paragraph
specifying error signature semantics. Concrete example:
`ConnectionTimeout{after: 30s}` and `ConnectionTimeout{after: 45s}`
share a signature; `ConnectionTimeout` and `ConnectionRefused`
pin independently.

---

## Q7. Provider Plugin Path — CONFIRMED NO CHANGE

**Decision:** Internal trait implementations only for v1. Users
with bespoke API formats fork the compiler.

**Rationale:** §17.8 already states this correctly: "Additional
providers can be added as trait implementations without changes
to the codegen pipeline. The interface is designed so that two
implementations cover the current provider landscape; a plugin
system is not warranted at this time."

Phase D implements the provider factory as a static match on
provider string. No plugin loading infrastructure required.
