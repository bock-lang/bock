# §19.7 stability tiers and §20.6 build features deferred

**Date:** 2026-05-13
**Affects:** §19.7, §20.6
**Type:** deferral / clarification
**Part of:** D1+D2 spec alignment consolidation

## Summary

§19.7's versioning stability tiers (alpha, beta, stable) are deferred to v1.x: the tier scheme presupposes an ecosystem of third-party packages, which v1 does not yet have. §20.6's introductory feature list (remote cache, build hooks, distributed builds) is qualified to match Appendix A.3's existing deferrals of the corresponding configuration surfaces. Both are editorial alignments with prior amendments that left this prose unupdated.

## Change

### §19.7 Stability tiers

Current §19.7 describes a three-tier stability scheme (alpha, beta, stable) for packages in the manifest, with `production-strictness` mode rejecting non-stable dependencies. The tier field is not in the manifest schema today, and rejection logic is not implemented.

Add a leading note to §19.7:

> **Status:** Deferred to v1.x. Stability tiers become useful once an ecosystem of third-party Bock packages exists. v1 ships with first-party stdlib packages only; the tier scheme and the production-strictness rejection logic are deferred to v1.x. The mechanism described below is preserved as design intent.

The §19.7 body text remains as the planned v1.x mechanism.

### §20.6 Build features

Current §20.6 opens with a brief feature list mentioning remote cache, build hooks, and distributed builds before transitioning to material that ships in v1. The Appendix A.3 Reserved markers have already deferred the corresponding `[build.cache]` (remote), `[build.hooks]`, and distributed-builds configuration surfaces. The §20.6 introduction needs to match.

Qualify the §20.6 opening sentence:

> Bock's build system supports incremental compilation, parallel builds across packages, and per-target output isolation as described in §20.6.1 and §20.6.2. Additional capabilities including remote cache reuse, build hooks, and distributed builds are deferred to v1.x; their configuration surfaces are marked Reserved in Appendix A.3.

The rest of §20.6 (which describes the v1 build system: output layout, output modes, source mapping, target profiles) is unchanged.

## Rationale

**§19.7 alignment.** Stability tiers without an ecosystem of packages are scaffolding that does not get used. The mechanism is sensible, but it is a v1.x feature once third-party packages exist. The Appendix A.3 deferrals already reflect this for the manifest schema; §19.7's prose was left describing the tier scheme as if v1.

**§20.6 alignment.** The feature list mentions capabilities that v1 does not implement (remote cache, build hooks, distributed builds). Appendix A.3 has marked their configuration surfaces Reserved. The §20.6 introduction implies these are v1 features, which contradicts A.3.

Both changes are editorial: matching prose to existing deferrals. No new design decisions are introduced.

- §20.6: incremental and parallel builds work; remote cache, hooks, distributed builds are not implemented
