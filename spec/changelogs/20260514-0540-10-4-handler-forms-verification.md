### Form C's spec status

`spec/bock-spec.md` §10.4 contains exactly two handler examples:

- **Form A** at lines 856–864: `record ConsoleLog {} + impl Log for ConsoleLog { fn log(...) ... }`
- **Form B** at lines 868–872: `let silent = Log.handler(log: (level, message) => {})` (multi-line, named-field lambda constructor)

Form C — `<Effect>.handler((args) => ...)` with no field name — does
**not** appear anywhere in `spec/bock-spec.md`. Grep confirms:

```
$ grep -nE 'handler\(' spec/bock-spec.md
869:let silent = Log.handler(
$ grep -nE 'handler\s*\(\s*\(' spec/bock-spec.md      # bare-lambda shape
  (no matches)
```

The only `.handler(` occurrence in the spec is the named-field
form on line 869. The session prompt's "Form C" is therefore an
implementation-chat construct (a sibling form posed as a question)
rather than a spec-documented surface.

### Form A — Trait impl block

**Verification path:** Strategy 1 (test corpus heavily exercises Form A) + Strategy 3 (minimal scratch program).

**Findings:**

- **Parses:** Yes. `parse_impl_block` accepts `impl Trait for Type { fn op(...) ... }`.
- **Type-checks:** Yes. `traits::resolve_impl` looks up the `(trait, type)` pair in the `ImplTable` built during checking; the type checker's effect inference subtracts effects supplied by `HandlingBlock` handlers in scope.
- **Dispatches:** Yes. The interpreter's `dispatch_effect_op` calls `try_call_impl_method` (interp.rs:2223) which finds the operation by name in the impl's method table and invokes it as a `self`-instance method.

**Source citations:**

- AST: `compiler/crates/bock-ast/src/lib.rs:966` — `pub struct ImplBlock`
- Parser: `compiler/crates/bock-parser/src/lib.rs:571` (`Item::Impl(self.parse_impl_block(...))`); `compiler/crates/bock-parser/src/lib.rs:4144` — `fn parse_impl_block`; `compiler/crates/bock-parser/src/lib.rs:4178` — `fn parse_impl_header` (recognizes `impl Trait for Type`)
- Type check / impl table: `compiler/crates/bock-types/src/traits.rs:165` (`NodeKind::ImplBlock` indexed); `compiler/crates/bock-types/src/traits.rs:386` — `pub fn resolve_impl`

- Dispatch: `compiler/crates/bock-interp/src/interp.rs:605` (`HandlingBlock` pushes handler value into the effect stack frame); `compiler/crates/bock-interp/src/interp.rs:649` — `dispatch_effect_op`; `compiler/crates/bock-interp/src/interp.rs:2223` — `try_call_impl_method`

### Form B — Named-field lambda constructor

**Verification path:** Strategy 1 (grep — zero occurrences in `compiler/tests/`, `examples/`, `stdlib/`) + Strategy 2 (source inspection: no dedicated parser / AST / checker support) + Strategy 3 (scratch program).

**Findings:**

- **Parses:** Yes — but only as a *generic* postfix method-call expression. The parser does not recognize `Effect.handler(...)` as a distinct construct. `Logger.handler(log: (msg) => ...)` is accepted as the same shape as any other `recv.method(named: arg)` expression.
- **Type-checks:** No. The receiver `Logger` is treated as an `Identifier` reference in the value namespace. Effect names are not bound in the value environment, so `env.lookup("Logger")` returns `None` and the checker emits `E4002 undefined variable \`Logger\`` (checker.rs:1239–1255). Inference never reaches the `.handler(...)` call.
- **Dispatches:** No — never reached. (See the adjacent observation below: the interpreter's `dispatch_effect_op` *would* accept a record-with-lambda-fields value if the front end produced one, but no front-end path constructs such a value.)

**Source citations:**

- Parser (generic postfix path used, no Form B specialization): `compiler/crates/bock-parser/src/lib.rs:1233` — `fn parse_postfix`; `:1287` — `Expr::MethodCall { ... }`
- AST (no dedicated handler-constructor node — only the generic `Expr::MethodCall`): `compiler/crates/bock-ast/src/lib.rs` — search for `Effect.handler` returns zero matches; no `HandlerExpr` / `EffectHandlerCtor` node type exists
- Type check (name resolution failure): `compiler/crates/bock-types/src/checker.rs:1239` — `NodeKind::Identifier { name }` → `env.lookup`; `:1247–1252` — `E_UNDEFINED_VAR` emission
- Diagnostic catalog: `compiler/crates/bock-errors/src/catalog.rs:217` — `code: "E4002"`
- Dispatch (would-be path, not reached): `compiler/crates/bock-interp/src/interp.rs:658–668` (Record-with-operation-field branch of `dispatch_effect_op`)
- Test usage: **none.** `grep -rnE '[A-Z][a-zA-Z]*\.handler\s*\(' compiler/tests/ examples/ stdlib/` returns zero matches.
- Confirmation run on `/tmp/$BOCK_TEST_NAMESPACE/form-b.bock`:

    at form-b.bock:12:11

### Form C — Bare-lambda constructor

**Verification path:** Strategy 2 (Form C is identical to Form B at the parser/checker layer) + Strategy 3 (scratch program).

**Findings:**

- **Parses:** Yes — as a generic postfix method-call expression, exactly as Form B.
- **Type-checks:** No. Same failure mode as Form B: the `Logger` receiver fails name resolution before the call is inferred. `E4002 undefined variable \`Logger\``.
- **Dispatches:** No — never reached. The interpreter's `dispatch_effect_op` *would* accept a `Value::Function` directly via the plain-function branch (interp.rs:671–674) — Form C's semantic counterpart already exists in the runtime — but no front-end path constructs it.

**Source citations:**

- Parser: same `parse_postfix` / `Expr::MethodCall` path as Form B
- AST: same `Expr::MethodCall` representation as Form B; no dedicated node
- Type check: same `E4002` site in `checker.rs:1247`
- Dispatch (would-be path, not reached): `compiler/crates/bock-interp/src/interp.rs:670–674` — `Value::Function` branch of `dispatch_effect_op` (single-operation effects can dispatch a bare function)
- Spec: Form C is absent from `spec/bock-spec.md` (confirmed via `grep -nE 'handler\s*\(\s*\(' spec/bock-spec.md` — zero matches)

## Implementation chat's recommendation

Verification matches the first shape posed in the session prompt:
**only Form A works.** Forms B and C share the same failure mode
(name resolution rejects `<Effect>.<member>` because effect names
are not bound in the value namespace) and there is no dedicated
parser / AST / checker path for either.

The implication for the upcoming `20260513-0515-specs-changes.md`
finalization is therefore:

- The §10.4 deferral language should mark **both** Form B and
  Form C as Reserved for v1.x. The current 0515 draft text only
  calls out the `Effect.handler(...)` form generically — which is
  technically correct (both shapes route through the same
  `Effect.handler(...)` syntax and both are deferred) but could
  be sharpened to acknowledge that Form C (`Effect.handler(lambda)`)
  was never in the spec to begin with, and would be a *new* surface
  added in v1.x rather than a deferred existing surface.
- The spec example at lines 868–872 (the Form B example) is the
  only example whose status changes: it is now explicitly marked
  as deferred design intent. The current 0515 draft already
  preserves it under a "Status:" caveat; that wording is
  appropriate.
- D2's FOUND row prose ("Not implemented; record-shaped handlers
  work") is consistent with verification. "Record-shaped handlers"
  refers to Form A (record + impl block) — the only handler shape
  the docs page demonstrates and the only shape that works.

Design chat owns whether to call out Form C as a separate v1.x
addition versus folding it into the same deferral entry as Form B.

## Adjacent observations

These are findings surfaced during verification that don't bear
directly on the §10.4 deferral language but design chat should
be aware of:

1. **Runtime supports more handler shapes than the front end.**
   `compiler/crates/bock-interp/src/interp.rs:649–678`
   (`dispatch_effect_op`) accepts three handler-value shapes:
   - A `Value::Record` whose fields include the operation name
     bound to a function (would make inline-record-literal handlers
     work, e.g., `{ log: (msg) => ... }` — see observation #2)
   - A `Value::Record` with an impl-block method matching the
     operation name (Form A — currently the only reachable path)
   - A `Value::Function` — a plain function, dispatched directly
     for single-operation effects (Form C's semantic counterpart)

   If the front end were updated to (a) treat
   `<Effect>.handler(...)` as a dedicated constructor that
   synthesizes the appropriate value, or (b) accept inline record
   literals as handler expressions, the runtime would already
   dispatch them correctly. The front end is the binding constraint
   for both Forms B and C, not the runtime.

2. **The 0515 draft's "record-shaped form" example does not parse.**
   `20260513-0515-specs-changes.md` lines 46–48 show:

   ```bock
   with Logger as { log: (msg) => print(msg) }
   ```

   as the "record-shaped form" that "works today". Verified
   against the current compiler:
   - The `with X as ...` *statement* syntax is not in the parser
     (`E2020 expected expression, found \`with\``). Bock's effect
     handler installation syntax is `handling (Effect with handler)
     { ... }`, not `with Effect as ...`.
   - Even rewritten as `handling (Logger with { log: (msg) => print(msg) })
     { ... }`, the inline record literal fails: the parser treats
     `log: ...` as a name reference rather than a structural field
     binding and emits `E1001 undefined name \`log\``.

   The 0515 draft's claim that record-shaped handlers work today
   is correct for Form A (records-with-impl-blocks) but is
   illustrated with an example surface that doesn't parse. Design
   chat may want to either drop the example or replace it with a
   working Form A snippet.

3. **`bock check` exit code does not reflect type errors.** All
   three forms' check runs printed `E4002` / `E2020` errors to
   stderr but returned exit code 0. This is independent of the
   §10.4 verification (Form A correctly exits 0 with no errors;
   Forms B/C "incorrectly" exit 0 despite errors). Likely a CLI
   bug worth filing separately — it does not affect this
   verification's conclusions because the diagnostic text was
   used as evidence rather than the exit code.

4. **No conformance fixture covers any effect handler dispatch.**
   `compiler/tests/conformance/effects/` contains a single fixture
   (`pure_function.bock`) that exercises `pure fn` rather than
   handler dispatch. End-to-end coverage of Form A is provided
   only by the `examples/` tree (notably
   `examples/spec-exercisers/effect-showcase`). Design chat may
   want a follow-up to surface this as a coverage gap for the
   implementation chat — not in scope for this verification but
   worth flagging.
